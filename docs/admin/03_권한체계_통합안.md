# 권한체계 통합안

> 목적: 관리자 웹과 회원앱 권한 체계를 분리 정리해, 화면 노출과 액션 제한 기준을 고정한다.

---

## 1. 기본 원칙

- 관리자 웹 권한과 회원앱 역할은 분리한다.
- 같은 사용자가 두 채널에 모두 접근하더라도, 화면 설계 기준은 채널별로 따로 본다.
- 퍼블리싱에서는 "보인다/숨긴다/조회만 가능/수정 가능" 기준이 명확해야 한다.

---

## 2. 관리자 웹 권한 체계

### 2.1 표준 역할

| 역할 키 | 표시명 | 설명 |
|------|------|------|
| `superAdmin` | 슈퍼관리자 | 본사 운영 총괄, 전 지점 접근 |
| `primary` | 최고관리자 | 지점 최상위 운영 권한 |
| `owner` | 센터장 | 지점 경영/운영 |
| `manager` | 매니저 | 운영 실무 총괄 |
| `fc` | FC | 상담/회원관리/일부 결제 |
| `staff` | 스태프 | 조회/현장 처리 |
| `readonly` | 조회전용 | 읽기 전용 |

### 2.2 비표준/정리 대상 역할

| 표현 | 처리 기준 |
|------|------|
| `front` | 관리자 웹에서는 `staff`에 통합 |
| `trainer` | 관리자 웹 표준 역할로 사용하지 않음 |
| `admin` | 문맥 모호. `primary` 또는 `superAdmin` 중 하나로 명시 치환 |

### 2.3 관리자 웹 역할 운영 원칙

- `superAdmin`만 본사 전용 화면 접근 가능
- `readonly`는 조회만 가능하며 상태 변경, 저장, 삭제, 발송, 승인 불가
- `fc`는 회원/리드/재등록 상담 중심
- 결제는 `manager` 이상 전권, `fc`는 현장 예외 화면에서만 허용 가능

---

## 3. 회원앱 역할 체계

### 3.1 표준 역할

| 역할 키 | 표시명 | 설명 |
|------|------|------|
| `member` | 회원 | 셀프서비스 사용자 |
| `trainer` | 트레이너 | 수업/회원 기록 담당 |
| `golf_trainer` | 골프강사 | 트레이너 확장 + 쌍방서명 |
| `fc` | FC | 상담/재등록 담당 |
| `staff` | 스태프 | 현장 조회/수동 출석 |

### 3.2 회원앱 원칙

- 회원앱은 역할 기반 통합 모바일 앱으로 정의한다.
- 로그인 후 role에 따라 다른 Navigator로 분기한다.
- "CRM 앱 별도 진입" 같은 문구는 회원앱 통합 전략과 충돌하므로 제외한다.

### 3.3 단일 역할 원칙

- 회원앱은 한 계정 1역할을 기본 원칙으로 둔다.
- 같은 사람이 회원과 트레이너를 모두 수행한다면 별도 계정 또는 별도 role entitlement로 처리한다.

---

## 4. 채널 분리 기준

| 구분 | 관리자 웹 | 회원앱 |
|------|------|------|
| 목적 | 운영/관리 | 현장/회원 모바일 사용 |
| 권한 키 | `superAdmin` ~ `readonly` | `member` ~ `staff` |
| 공통 데이터 | 공유 | 공유 |
| 공통 내비 | 없음 | 없음 |
| 화면 기준 | 데스크톱 Backoffice | 모바일 통합 앱 |

---

## 5. 화면 퍼블리싱 기준

### 5.1 관리자 웹

- 버튼 비활성보다 라우트 차단이 맞는 화면과, 조회는 허용하고 액션만 비활성인 화면을 구분한다.
- 예:
  - 본사 전용 화면: 진입 차단
  - 회원 상세 읽기전용: 진입 허용, 액션 숨김/비활성

### 5.2 회원앱

- 공통 로그인/온보딩은 전 역할 공통
- 로그인 성공 후 홈/탭 구성은 role별 완전 분기
- 회원앱의 `staff`, `fc`, `trainer`, `golf_trainer`는 모바일 셸을 공유하는 운영 역할로 본다

## 7. 정리 대상 문구

아래 표현은 추후 원본 문서 정비 시 정리 대상이다.

- `trainer`가 관리자 웹 역할처럼 등장하는 문구
- `front`를 별도 표준 권한처럼 쓰는 문구
- 회원앱에서 `staff/admin은 별도 CRM 앱 사용`으로 적힌 문구
- `슈퍼관리자/primary`를 같은 의미로 섞어 쓰는 문구

---

## 8. 결론

권한 체계는 "누가 어떤 버튼을 보느냐"를 정하는 기준이므로, 상태 체계만큼 먼저 고정되어야 한다.

이후 문서 정비에서는 아래를 강제한다.

1. 관리자 웹은 `superAdmin` 체계를 사용
2. 회원앱은 `member/trainer/golf_trainer/fc/staff` 체계를 사용
3. 채널 간 역할 이름 혼용 금지
