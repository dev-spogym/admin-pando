# 자동화 / 만료 알림 정책 개정안

> 목적: 만료 알림을 고정 D-day 세트가 아니라, 본사가 정책 세트와 step을 등록하고 지점이 그 기준에 따라 적용 정책을 수립하는 구조로 재설계한다.

---

## 1. 개정 배경

현재 문서군은 아래처럼 서로 다른 만료 알림 기준을 전제로 한다.

- `D-30 / D-14 / D-7 / D-3 / D-1`
- `D-30 / D-7 / D-1`
- `D-7 / D-3 / D-1`

이 방식은 아래 문제가 있다.

1. 본사가 운영 품질을 통제하기 어렵다.
2. 지점별 운영차를 반영하기 어렵다.
3. 상품 유형별로 다른 리텐션 전략을 적용하기 어렵다.
4. 퍼블리싱 기준 화면에서 어떤 step이 노출되는지 고정하기 어렵다.

따라서 만료 알림은 "고정 D-day"가 아니라 "본사 제공 정책 세트" 기반으로 운영한다.

---

## 2. 목표 구조

### 최종 구조

1. 본사가 만료 알림 정책 세트와 step 기준을 등록한다.
2. 정책 세트 안에는 여러 알림 step과 기본 채널/메시지 규칙이 들어간다.
3. 각 지점은 본사가 제공한 정책 세트를 기준으로 자기 지점의 적용 정책을 수립한다.
4. 지점은 허용된 범위에서 step on/off, 채널 사용, 후속 액션 연결만 조정한다.
5. step의 기준일, 중요도, 기본 메시지는 본사 기준을 따른다.
6. 지점은 본사가 등록하지 않은 새로운 기준일 step을 임의로 만들지 않는다.

---

## 3. 운영 원칙

### 3.1 본사

- 정책 세트를 생성/수정/종료할 수 있다.
- step별 기준일, 채널, 발송 시각, 메시지 유형을 정의한다.
- 특정 step을 `필수`, `권장`, `선택`으로 지정할 수 있다.
- 정책을 상품군 또는 센터 유형별로 배포할 수 있다.

### 3.2 지점

- 본사 제공 정책 세트만 선택할 수 있다.
- 본사가 등록한 step 기준을 바탕으로 지점 적용 정책을 수립한다.
- `선택 가능`으로 열린 step만 활성/비활성 고를 수 있다.
- `필수` step은 끌 수 없다.
- 자체적으로 새로운 기준일 step을 생성하지 않는다.

### 3.3 시스템

- 매일 정책 세트와 지점 적용값을 읽는다.
- 각 이용권/락커/수강권에 대해 정책 step 조건을 평가한다.
- 이미 발송된 step은 중복 발송하지 않는다.

---

## 4. 적용 대상

정책은 아래 범위별로 운영 가능하다.

| 범위 | 설명 |
|------|------|
| 회원권 | 일반 기간제 이용권 |
| PT / 횟수권 | 횟수제 수강권 |
| 골프 레슨권 | 골프 수강권 |
| 락커 | 시설 사용권 |
| 운동복 / 부가시설 | 선택 적용 |

본사 정책은 범위별로 별도 세트 운영이 가능해야 한다.

## 5. 정책 모델

## 5.1 본사 정책 세트

| 항목 | 설명 |
|------|------|
| 정책 세트 ID | 시스템 고유값 |
| 정책명 | 예: 표준 회원권 리텐션형 |
| 적용 범위 | 회원권 / PT / 골프 / 락커 |
| 적용 대상 | 전체 지점 / 일부 지점군 / 브랜드군 |
| 상태 | 초안 / 배포중 / 종료 |
| 설명 | 운영 목적 및 사용 가이드 |
| 기본 여부 | 해당 범위의 기본 추천 세트 여부 |

## 5.2 정책 Step

| 항목 | 설명 |
|------|------|
| Step ID | 정책 세트 내부 고유값 |
| 기준일 타입 | 만료 전 / 만료 당일 / 만료 후 |
| 기준값 | N일 |
| 채널 | 앱푸시 / 카카오 / SMS / 이메일 |
| 발송 시각 | 예: 09:00 |
| 제목 템플릿 | 메시지 제목 |
| 본문 템플릿 | 메시지 본문 |
| 중요도 | 필수 / 권장 / 선택 |
| 지점 제어 가능 여부 | on/off 허용 여부 |
| 중복 발송 제한 | 회원+상품+step 기준 1회 |

## 8. 지점 화면 기획

### 화면명

- `설정관리 > 자동 알림 설정 > 만료 알림 정책 선택`

### 화면 목적

- 지점이 본사 제공 정책 세트를 확인하고, 자기 지점에 적용할 정책을 수립한다.

### 주요 구성

1. 현재 적용 정책 카드
2. 사용 가능한 정책 세트 목록
3. 선택 가능한 step 토글 리스트
4. 예상 발송 미리보기
5. 적용 대상 상품군 선택

### 지점 선택 규칙

- 범위별로 1개의 활성 정책 세트만 선택 가능
- `필수` step은 비활성 불가
- `권장/선택` step만 on/off 가능
- 기준일 값은 수정 불가
- 기준일 값은 본사가 입력한 step을 그대로 따른다
- 채널 수정은 본사가 허용한 경우에만 가능

### 지점 화면 와이어프레임


## 10. 자동화 처리 규칙

### 10.1 일 배치 처리 순서

1. 활성 지점 정책 조회
2. 범위별 적용 정책 확인
3. step별 기준일 계산
4. 대상 회원/이용권 조회
5. 중복 발송 여부 확인
6. 채널별 발송 큐 적재
7. 발송 결과 기록

### 10.2 중복 방지 키

- `branchId + entityType + entityId + policyId + stepId + targetDate`

### 10.3 발송 조건

- 회원 수신 동의 여부 충족
- 이용권 상태가 발송 대상 상태여야 함
- 이미 같은 step이 발송되지 않았어야 함
- 지점이 해당 step을 활성화했어야 함

## 12. 예외 처리

| 상황 | 처리 기준 |
|------|------|
| 본사 정책 종료 | 지점에 종료 예정 경고 후 다른 정책 선택 유도 |
| 지점 미선택 | 해당 범위 자동화 중지 + 관리자 경고 |
| 필수 step 채널 오류 | 대체 채널 우선순위 적용 또는 실패 로그 적재 |
| 메시지 템플릿 수정 중 | 배포중 정책은 새 버전으로 복제 후 수정 |
| 동일 회원 다중 상품 | 상품 범위별로 각 정책 평가, 단 동일 채널 과다발송은 묶음 안내 가능 |

## 13. 수용 기준

- 본사는 범위별 정책 세트를 여러 개 등록할 수 있어야 한다.
- 지점은 본사가 배포한 정책만 선택할 수 있어야 한다.
- 지점은 허용된 step만 끄고 켤 수 있어야 한다.
- 필수 step은 지점이 끌 수 없어야 한다.
- 배치는 본사 등록 step과 지점 적용 정책을 기준으로 대상과 기준일을 계산해야 한다.
- 발송 이력은 정책 세트와 step 기준으로 추적 가능해야 한다.
- 퍼블리싱 화면에서는 정책 소유권이 항상 "본사"인지 "지점"인지 명확히 보여야 한다.

---

## 14. 결론

이 개정안으로 만료 알림 정책은 아래처럼 정리된다.

- 본사: 전략 설계자
- 지점: 적용 정책 수립자
- 시스템: 정책 실행자

즉, 지금처럼 문서마다 서로 다른 고정 D-day를 전제하는 구조를 버리고, 본사가 등록한 step과 정책 세트를 기준으로 지점이 적용 정책을 수립하는 구조로 통합한다.
