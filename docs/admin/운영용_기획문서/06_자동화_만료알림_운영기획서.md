# 자동화 · 만료알림 운영기획서

## 1. 문서 목적

- 자동화와 만료 알림을 고정 규칙이 아니라 운영 가능한 정책 체계로 정리한다.
- 특히 만료일 알림은 본사가 기준 step을 만들고 지점이 그 기준에 따라 적용 정책을 수립하는 구조를 명확히 한다.

## 2. 핵심 결론

- 기존의 고정 `D-30 / D-14 / D-7 / D-3 / D-1` 전제는 운영 기준으로 사용하지 않는다.
- 본사가 만료 알림 정책 세트를 등록한다.
- 지점은 본사가 만든 정책 세트를 기준으로 자기 지점 적용 정책을 수립한다.
- 지점은 본사가 허용한 범위 안에서만 스텝 on/off 또는 채널 사용 여부를 조정할 수 있다.
- 지점이 본사 미등록 기준일 step을 임의로 만드는 구조는 기본값으로 허용하지 않는다.

## 3. 자동화 대상

- 정기 이용권 만료 임박
- PT 이용권 잔여 회차 부족 또는 만료 임박
- 골프 이용권 만료 임박
- 락커/부가서비스 만료 임박
- 예약 확정/변경/취소
- 노쇼 경고
- 생일/휴면/재방문 유도

## 4. 자동화 정책 구조

### 4.1 정책 세트

| 항목 | 설명 |
| --- | --- |
| 정책명 | 본사가 구분하는 운영용 이름 |
| 적용 범위 | 정기권, PT, 골프, 락커 등 |
| 사용 상태 | 사용중, 중지 |
| 기본 여부 | 지점 추천 기본값 여부 |
| 지점 수정 허용 | 지점이 일부 옵션 변경 가능한지 여부 |

### 4.2 정책 스텝

| 항목 | 설명 |
| --- | --- |
| 기준일 | 본사가 등록한 만료일 기준 상대 일수 |
| 스텝명 | 예: 1차 안내, 재등록 제안 |
| 채널 | 앱푸시, SMS, 알림톡, 이메일 |
| 메시지 템플릿 | 본사 등록 템플릿 |
| 필수 여부 | 필수, 권장, 선택 |
| 지점 토글 허용 | 지점에서 끌 수 있는지 여부 |

## 5. 운영 원칙

### 5.1 본사 원칙

- 본사는 전사 기준 정책 세트를 등록한다.
- 본사는 상품군별 권장 정책을 정한다.
- 본사는 알림 문구 템플릿과 채널 우선순위를 정한다.
- 본사는 지점이 수정 가능한 범위를 설정한다.

### 5.2 지점 원칙

- 지점은 정책 세트를 새로 만들지 않고, 본사 등록 step을 기준으로 적용 정책을 수립한다.
- 지점은 허용된 스텝만 비활성화할 수 있다.
- 지점은 본사가 허용한 경우에 한해 발송 채널만 조정할 수 있다.
- 지점별 담당자 자동 배정이나 후속 액션 연결은 지점 운영 규칙으로 둔다.

- D-1: 마지막 안내

## 7. 만료 기준 계산 원칙

- 만료 기준은 실제 조정된 만료일을 따른다.
- 홀딩이 있으면 홀딩 반영 후 만료일을 기준으로 재계산한다.
- 환불/취소로 종료된 이용권은 이후 만료 알림 대상에서 제외한다.
- 이미 재등록이 완료된 경우 이전 이용권 기준 알림은 중단한다.

## 8. 화면 운영 기준

### 8.1 본사 화면

- 화면명: 자동화 정책 라이브러리
- 필요한 영역
  - 정책 세트 목록
  - 정책 상세
  - 적용 범위
  - 스텝 편집
  - 템플릿 선택
  - 지점 수정 허용 범위 설정

### 8.2 지점 화면

- 화면명: 자동화 적용 설정
- 필요한 영역
  - 상품군별 적용 정책 수립
  - 허용된 스텝 토글
  - 발송 채널 확인
  - 담당자 후속 액션 연결
  - 테스트 발송

## 9. 운영 흐름

```text
본사에서 정책 세트와 step 등록
-> 지점에서 상품군별 적용 정책 수립
-> 대상 회원 자동 산출
-> 스텝별 발송
-> 발송 이력 저장
-> 상담/재등록 후속 액션 연결
```

## 10. 운영 예외 처리

- 수신 동의가 없는 채널은 자동 제외한다.
- 동일 회원에게 같은 날 중복 성격의 알림이 많으면 묶음 처리 기준을 둔다.
- 담당 FC가 없는 회원은 지점 공용 큐로 보낸다.
- 지점이 정책을 바꾸면 변경 시점 이후 대상부터 새 정책을 적용한다.

## 11. 회원앱/관리자웹 연계 기준

- 회원앱 알림 목록은 자동화 발송 이력과 일관되어야 한다.
- 관리자 웹에서는 발송 여부뿐 아니라 후속 처리 상태까지 봐야 한다.
- 만료 알림은 단순 발송으로 끝나지 않고 상담/결제 액션 큐와 연결되어야 한다.

## 12. 퍼블리싱 기준 메모

- 본사 화면은 "정책 설계"가 중심이다.
- 지점 화면은 "무엇을 적용 중인지"가 중심이다.
- 고정 D-day UI가 아니라 정책 스텝 리스트 UI로 설계해야 한다.
