각 전환별 확인 메시지:
  active → maintenance:
    title: "{roomName}을 점검 처리하시겠습니까?"
    description: "점검 중에는 수업 예약이 불가합니다."
    confirmLabel: "점검 처리"
    variant: "warning"

  maintenance → broken:
    title: "{roomName}을 고장 처리하시겠습니까?"

    confirmLabel: "고장 처리"
    variant: "danger"

  broken → active:
    title: "{roomName} 운영을 재개하시겠습니까?"

    confirmLabel: "운영 재개"
    variant: "primary"

handleStatusChange(room):
  const nextStatus = { active: 'maintenance', maintenance: 'broken', broken: 'active' }[room.status]
  await supabase.from('rooms').update({ status: nextStatus }).eq('id', room.id)
  const messages = {
    maintenance: `${room.name}이 점검 처리되었습니다.`,
    broken: `${room.name}이 고장 처리되었습니다.`,
    active: `${room.name} 운영이 재개되었습니다.`,
  }
  toast.success(messages[nextStatus])
  fetchRooms()

| 블록 | 테이블 | 조건 |
|------|--------|------|

### 인터랙션 (User Actions)
1. [상태 변경] → 확인 다이얼로그 오픈
2. [확인] → 상태 순환 전환 + toast
3. [취소] → 다이얼로그 닫기

### 비즈니스 룰
- 상태 순환: `active → maintenance → broken → active`
- `maintenance` / `broken` 상태에서 수업 예약 불가 (수업관리 도메인에서 처리)
- 고장 처리 시 해당 날짜 이후 예약 자동 취소 여부 별도 정책 확인 필요
- 상태 변경은 센터장/매니저만 가능 (스태프는 버튼 비활성)

### 에지 케이스
- 예약이 있는 룸 → 점검/고장 처리 시 "예약된 수업 N건이 있습니다." 추가 경고
