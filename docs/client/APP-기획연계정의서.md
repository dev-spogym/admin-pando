## 3. 이번 정리의 핵심 원칙

### 3.1 회원가입

- 회원 기본 생성 주체는 `CRM`이다.
- 앱에서는 `휴대폰 인증 + 비밀번호 설정`으로 연동/가입을 완료한다.
- 즉, 체크리스트의 `회원가입`은 현재 기준에서 `CRM 선등록 + 모바일 앱 연동`으로 해석한다.
- 공개형 즉시 회원 생성은 현재 기본 정책이 아니다.

### 3.2 로그인 구분

- 공통 로그인 화면 `MA-001`은 `회원 로그인`과 `직원 로그인` 두 진입으로 해석한다.
- 회원 로그인은 `전화번호 + 비밀번호`를 사용한다.
- 직원 로그인은 `아이디 + 비밀번호`를 사용하며, 로그인 전 `트레이너`, `FC`, `스태프` 중 직원 유형을 먼저 선택한다.
- `golf_trainer` 계정은 직원 로그인 화면에서는 `트레이너` 계열로 묶는다.
- 선택한 직원 유형과 실제 역할이 다르면 로그인 실패 처리한다.
- 로그인 성공 후에는 회원, 트레이너 계열, FC, 스태프 각 모드로 분기한다.
- 회원앱 role 키는 `member / trainer / golf_trainer / fc / staff`를 기준으로 해석한다.
- 이 role 체계는 관리자웹 권한 키(`superAdmin / primary / owner / manager / fc / staff / readonly`)와 별도로 운영한다.
- 같은 표기인 `fc`, `staff`도 채널에 따라 접근 범위와 화면이 다르며, 이 문서에서는 회원앱 role로만 해석한다.

### 3.3 리워드

- 체크리스트의 `포인트`는 `마일리지`로 통일한다.
- `포인트 사용`은 `마일리지 사용`으로 해석한다.
- `배지`는 후속이 아니라 현재 회원앱 기획 범위에 포함한다.
- 따라서 리워드 체계는 `쿠폰 + 마일리지 + 배지` 3축으로 정의한다.

### 3.4 온보딩

- 체크리스트의 `운동 목적/성향`, `통증/체형 정보`, `첫 루틴 제안`은 현재 회원앱 범위에 포함한다.
- 앱 연동 직후 또는 첫 로그인 직후 회원이 온보딩을 진행한다.
- FMS/체형 리포트는 온보딩 입력값과 측정 기록을 함께 참조한다.

### 3.5 문서 운영 원칙

- 문서는 기획과 퍼블리싱 기준을 우선한다.
- 상태값, 용어, 화면 흐름은 문서 기준으로 먼저 고정한다.
- 개발 구현 여부와 무관하게 회원이 이해할 수 있는 퍼블리싱 흐름은 문서 범위에 포함한다.

### 3.6 결제 / 상품 확장

- 체크리스트의 결제 범위는 `PT 플랜`에만 한정하지 않는다.
- 헬스장 이용권, 골프장 이용권, 골프 레슨권, 재등록 상품, 개인 결제를 모두 회원앱 결제 범위에 포함한다.
- 실제 결제 연동 여부와 무관하게 `상품 스토어`, `상품 상세`, `결제`, `영수증`, `개인 결제` 화면은 퍼블리싱 기준으로 제공한다.

### 3.7 개인정보 / 동의 / 탈퇴

- 체크리스트의 `개인정보·로그아웃`은 설정 한 화면이 아니라 `약관/정책`, `동의관리`, `회원 탈퇴`, `로그아웃`까지 포함한 묶음으로 해석한다.
- 회원 탈퇴는 현재 단계에서 `탈퇴 요청 접수` 화면으로 정의한다.

### 3.8 예약 주체 / 승인 정책

- 회원과 트레이너는 모두 수업 일정을 생성 또는 예약할 수 있는 주체로 본다.
- 회원이 트레이너의 가능 시간 또는 개별 PT 슬롯을 보고 예약하는 경우, 예약 상태는 `트레이너 승인 대기`를 기본으로 한다.
- 트레이너가 자신의 캘린더에서 회원을 지정해 수업을 직접 생성하는 경우, 별도 승인 절차 없이 즉시 확정으로 처리한다.
- 회원 문서에서는 `MA-121`, `MA-122`, `MA-123`, `MA-125`, `MA-131`과 `MFN-105`, `MFN-107`, `MFN-128`, `MFN-132`, `MFN-143~145`에 이 정책을 연결한다.
- 트레이너 문서에서는 `MA-210`, `MA-221`과 `MFN-202`, `MFN-210`, `MFN-211`, `MFN-217`, `MFN-223`, `MFN-224`에 같은 정책을 연결한다.

### 3.9 횟수 차감 기준

- PT 및 골프 레슨의 횟수 차감 시점은 `예약 승인`, `회원 직접 지정 생성`, `일정 확정`이 아니라 `수업 완료` 시점으로 해석한다.
- 회원과 트레이너가 보는 `잔여 횟수`는 완료 처리된 수업만 반영한 값이어야 한다.
- `다음 예약 예정 일자`는 차감 확정일이 아니라 향후 일정 참고 정보로 분리 표기한다.
- 따라서 문서에서 `차감 예정일`처럼 읽힐 수 있는 표현은 `최근 차감 일자 + 다음 예약 예정 일자` 구조로 해석한다.

### 3.10 만료 알림 정책 소유권

- 회원앱 문서에서는 만료 알림 기준일을 고정 `D-30/D-14/D-7/D-3/D-1` 규칙으로 확정하지 않는다.
- 만료 알림 기준은 A(관리자 웹)에서 관리하는 `본사 정책 세트 -> 지점 선택 적용` 구조를 따른다.
- 회원앱은 정책을 편집하지 않고, 현재 지점에 적용된 정책의 `step 결과`, `긴급도`, `남은 일수`를 표시하는 채널로 해석한다.
- 따라서 회원/FC 화면의 `D-7` 같은 문구는 표준 정책 예시 또는 계산 결과 예시로만 사용한다.

## 4. 체크리스트 용어 매핑

| 체크리스트 용어 | 회원앱 적용 용어 | 처리 기준 |
|---|---|---|
| 트레이너 로그인 | 직원 로그인 > 트레이너 선택 | 골프강사 포함 |
| 포인트 | 마일리지 | 동일 개념으로 통합 |
| 포인트 사용 | 마일리지 사용 | 결제 리워드 차감 의미 |
| 배지 | 배지 | 신규 포함 |
| 회원가입 | CRM 등록 + 앱 연동 | 휴대폰 인증 기반 |
| 온보딩 설문 | 운동 목적/성향 + 통증/체형 입력 | 신규 포함 |
| 첫 루틴 제안 | 온보딩 완료 결과 | 신규 포함 |
| 공지/이벤트 배너 | 홈 공지/이벤트 카드 | 공지 중심 + 이벤트 표현 확장 |
| FMS/체형 리포트 | 체성분/FMS 통합 화면 | 동일 도메인으로 묶음 |

| 이용권 | 회원권 만료/잔여회차 중심 | PT 차감 완료 이력, 잔여 횟수, 다음 예약 일정까지 포함 |
| 알림 | 문서상만 존재 | 회원 기준 실사용 화면으로 포함 |
| 온보딩 | 문서 누락 | 현재 범위 포함 |
| 재등록 | FC 중심 | 회원용 추천 플랜 화면 포함 |

## 6. 이번 정리 결과

- 체크리스트 37개 항목은 회원앱 IA, 화면설계서, 기능명세서에서 모두 추적 가능해야 한다.
- 로그인 구조는 `회원 로그인 / 직원 로그인`으로 통일하고, 직원 로그인 하위 역할은 `트레이너 / FC / 스태프`로 해석한다.
- `포인트`는 `마일리지`로 통일한다.
- `배지`, `알림센터`, `온보딩`, `대기예약`, `강사상세`, `수업후기`, `재등록 추천`은 회원앱 현재 기획 범위에 편입한다.
- 회원가입은 `CRM 등록 + 모바일 앱 연동`으로 정리한다.
- `골프 강사 예약`, `상품 스토어`, `개인 결제`, `결제`, `영수증`, `약관/동의/탈퇴`도 같은 기준으로 편입한다.
- 회원의 `내 골프 예약 일정`, `PT 잔여횟수 + 차감 완료 이력/다음 예약 일정`, `트레이너 스케줄 확인 후 예약 요청`, `트레이너 승인/직접 배정`도 현재 기획 범위에 편입한다.
- 트레이너의 `회원 예약 요청 승인/거절`, `담당 회원 수업 이력 조회`는 트레이너 문서와 연결해 동일 정책으로 관리한다.
