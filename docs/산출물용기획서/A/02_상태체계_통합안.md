# 상태체계 통합안

> 목적: 리드, 회원, 이용권, 결제 상태를 하나의 기준으로 정리해 화면 문구와 상태 전이를 통합한다.

---

## 1. 공통 원칙

- 상태는 `저장 상태`와 `표시 상태`로 나눈다.
- 저장 상태는 엔티티의 실제 생명주기를 표현한다.
- 표시 상태는 운영자가 빠르게 이해하도록 계산해 붙이는 배지/문구다.
- 표시 상태는 저장 상태를 덮어쓰지 않는다.

---

## 2. 리드 상태 통합안

### 2.1 최종 기준 상태

| 내부 기준 키 | 운영 표시명 | 설명 | 종료 여부 |
|------|------|------|------|
| `NEW` | 신규 | 최초 유입, 아직 실접촉 전 | 비종료 |
| `CONTACTED` | 연락완료 | 1차 접촉 완료, 추가 안내 가능 | 비종료 |
| `CONSULT_SCHEDULED` | 상담예정 | 상담/체험 방문 일정이 잡힘 | 비종료 |
| `VISITED` | 방문완료 | 방문 또는 상담 완료, 등록 검토 중 | 비종료 |
| `CONVERTED` | 등록완료 | 회원 전환 완료 | 종료 |
| `LOST` | 미전환 | 최종 전환 실패 | 종료 |
| `HOLD` | 보류 | 재연락 예정, 후속 조치 보유 | 비종료 |

### 2.2 운영 규칙

- `HOLD` 상태는 `followUpDate` 입력이 필수다.
- `CONVERTED` 시점에는 `convertedMemberId` 연결을 필수 프로세스로 본다.
- `CONVERTED`인데 `convertedMemberId`가 비어 있으면 경고 배지 `미연결`을 노출한다.

### 2.3 기존 문서 매핑

| 기존 표현 | 통합 상태 |
|------|------|
| 신규 | `NEW` |
| 연락완료 | `CONTACTED` |
| 상담예정 | `CONSULT_SCHEDULED` |
| 방문완료 | `VISITED` |
| 등록완료 | `CONVERTED` |
| 미전환 | `LOST` |
| 보류 | `HOLD` |
| INQUIRY | `NEW` |
| CONSULTING | `VISITED` 또는 `CONTACTED` 이후 진행 단계로 흡수 |
| RESERVED | `CONSULT_SCHEDULED` |
| PENDING | `HOLD` 또는 `CONTACTED` 문맥별 분리 사용 금지 |

### 2.4 퍼블리싱 기준

- 목록 필터와 칸반 컬럼은 한글 운영 표시명을 기준으로 통일한다.
- 칸반 기본 컬럼은 `신규 / 연락완료 / 상담예정 / 방문완료 / 등록완료` 5개를 기본으로 한다.
- `보류`, `미전환`은 별도 접이 영역 또는 상태 필터로 노출한다.

## 3. 회원 상태 통합안

### 3.1 저장 상태

| 저장 상태 | 표시명 | 설명 |
|------|------|------|
| `ACTIVE` | 활성 | 정상 이용 가능 |
| `INACTIVE` | 비활성 | 관리자 수동 비활성 처리 |
| `EXPIRED` | 만료 | 대표 이용권 기준 모든 유효 이용권 종료 |
| `HOLDING` | 홀딩 | 이용 중지 상태, 기간 연장 계산 대상 |
| `SUSPENDED` | 정지 | 이용 제한 |
| `WITHDRAWN` | 탈퇴 | 탈퇴 처리 완료 |
| `DORMANT` | 휴면 | 장기 미방문 |
| `TRANSFERRED` | 이관 | 타지점 이관 완료 |

### 3.2 표시 상태

| 표시 상태 | 설명 | 저장 여부 |
|------|------|------|
| `미등록` | 이용권 미보유 또는 아직 이용 시작 전 회원 표시용 상태 | 저장 안 함 |
| `만료임박` | 유효 이용권이 남아 있으나 만료 기준일 이내 | 저장 안 함 |
| `이용예정` | 결제 완료, 시작일 미도래 | 저장 안 함 |

### 3.3 핵심 정리

- `INACTIVE`는 `미등록`이 아니다.
- `WITHDRAWN`은 `INACTIVE`에 흡수하지 않는다.
- `DORMANT`와 `TRANSFERRED`는 별도 상태로 유지한다.
- 회원 목록의 `미등록` 탭은 `INACTIVE` 탭이 아니라 `미등록 표시 상태` 탭으로 재정의하는 것이 맞다.

### 3.4 회원 대표 상태 표시 우선순위

1. 정지
2. 탈퇴
3. 이관
4. 휴면
5. 홀딩
6. 활성
7. 만료
8. 비활성
9. 미등록 표시

### 3.5 퍼블리싱 기준

- 회원 상세는 `WITHDRAWN`, `DORMANT`, `TRANSFERRED`를 각각 별도 배너/읽기전용 정책으로 분기한다.
- `INACTIVE (비활성/탈퇴)` 같은 합쳐진 표기는 사용하지 않는다.
- `미등록`은 상태 배지가 아니라 보조 배지 또는 세그먼트 성격으로 다룬다.

## 4. 이용권 상태 통합안

| 상태 | 표시명 | 설명 |
|------|------|------|
| `SCHEDULED` | 이용예정 | 결제 완료, 시작일 미도래 |
| `ACTIVE` | 활성 | 사용 가능한 이용권 |
| `HOLDING` | 홀딩 | 홀딩 적용 중 |
| `EXPIRED` | 만료 | 기간 종료 또는 잔여 횟수 0 |
| `TRANSFERRED` | 양도 | 타 회원으로 이관 완료 |
| `REFUNDED` | 환불 | 환불 처리 완료 |

### 이용권 운영 규칙

- 회원 대표 상태보다 이용권 상태가 우선 기준이다.
- 만료 임박은 이용권 표시 계산 상태로만 사용한다.
- 기간권/횟수권은 같은 상태 체계를 공유하되 만료 조건만 다르게 계산한다.

---

## 5. 결제 상태 통합안

### 5.1 최종 기준 상태

| 내부 기준 키 | 표시명 | 설명 |
|------|------|------|
| `PENDING` | 결제대기 | 승인 대기 |
| `APPROVED` | 결제완료 | 정상 승인 |
| `INSTALLMENT` | 미수/분할진행 | 일부 금액 미수 또는 할부 진행 |
| `PARTIAL_REFUNDED` | 부분환불 | 일부 금액 환불 |
| `REFUNDED` | 전액환불 | 전액 환불 |
| `FAILED` | 결제실패 | 승인 실패 |
| `CANCELED` | 취소 | 승인 전 취소 |

### 5.2 기존 문서 매핑

| 기존 표현 | 통합 상태 |
|------|------|
| `COMPLETED` | `APPROVED` |
| `UNPAID` | `INSTALLMENT` |
| `PENDING` | `PENDING` |
| `REFUNDED` | `REFUNDED` |
| 일부결제 | `INSTALLMENT` |
| 연체 | `INSTALLMENT` 내부 세부 상태 또는 컬렉션 배지 |

### 5.3 퍼블리싱 기준

- 목록 기본 배지는 `결제완료 / 미수·분할진행 / 부분환불 / 전액환불 / 결제대기 / 실패 / 취소`만 사용한다.
- 기존 `완료/미납/환불/대기` 뷰는 상위 필터 alias로 유지 가능하다.
- 부분환불과 전액환불은 반드시 분리 표기한다.

---

## 6. 엔티티 간 연결 규칙

### 6.1 리드 -> 회원

- `CONVERTED`는 회원 생성 또는 기존 회원 연결이 완료되어야 한다.
- 실패 시 `CONVERTED`로 먼저 바꾸지 않고 `VISITED` 또는 `HOLD`에서 해결한다.

### 6.2 결제 -> 이용권

- `APPROVED` 직후 이용권 생성/활성화가 이어진다.
- `PENDING`이나 `FAILED` 상태에서는 이용권을 생성하지 않는다.

### 6.3 이용권 -> 회원 대표 상태

- 회원 상태는 이용권 묶음의 대표 계산 결과로 결정된다.
- 이용권이 없다고 회원 상태를 무조건 `INACTIVE`로 두지 않는다.

---

## 7. 즉시 적용해야 할 문구 치환

| 현재 자주 보이는 표현 | 치환 방향 |
|------|------|
| `INACTIVE = 미등록` | 분리 |
| `INACTIVE (비활성/탈퇴)` | 분리 |
| `COMPLETED/UNPAID`만 사용 | `APPROVED/INSTALLMENT` 중심으로 재해석 |
| `PENDING`을 보류/예약/상담대기 모두에 사용 | 금지 |
| `등록완료인데 미연결` 무시 | 경고 상태 추가 |

---

## 8. 결론

상태 체계가 흔들리면 화면 배지, 상태 필터, CTA, 빈 상태, 자동화 조건이 모두 흔들린다.

따라서 이후 퍼블리싱과 화면설계서 정비는 반드시 이 문서 기준으로 진행한다.
