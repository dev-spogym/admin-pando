# FN-003~007: 회원관리

## 기본 정보

### FN-003: 회원 목록 조회/검색/필터

| 항목 | 내용 |
|------|------|
| **기능 ID** | FN-003 |
| **기능명** | 회원 목록 조회/검색/필터 |
| **분류** | 회원관리 |
| **플랫폼** | Web |
| **우선순위** | Critical |
| **상태** | 확정 |
| **화면 ID** | SCR-010 |
| **관련 API ID** | API-007 |

#### 기능 설명
회원을 검색, 필터링, 정렬하여 조회하고 엑셀 다운로드 기능을 제공합니다. 300ms 디바운스로 서버 부하를 줄이고, 페이지당 건수 선택 옵션으로 유연한 조회를 지원합니다.

#### 상세 요구사항
1. 이름/연락처/회원번호 검색 (300ms debounce)
2. 상태/성별/담당자 필터 조합
3. 컬럼 정렬 (이름, 가입일, 상태 등)
4. 페이지당 20/50/100건 선택
5. 엑셀 다운로드 (현재 필터 결과)
6. 페이지네이션 (이전/다음/페이지 번호)
7. 선택 행의 일괄 작업 (선택사항)

#### 입력 항목

| 항목명 | 타입 | 필수여부 | 설명 |
|--------|------|---------|------|
| 검색어 | String | 선택 | 이름/연락처/회원번호 (300ms debounce) |
| 상태 | String[] | 선택 | "활성" \| "휴면" \| "만료" \| "탈퇴" |
| 성별 | String[] | 선택 | "남" \| "여" |
| 담당자 | UUID[] | 선택 | FC 담당자 ID (다중 선택 가능) |
| 페이지 | Number | 선택 | 기본값: 1 |
| 페이지당건수 | Number | 선택 | 20 \| 50 \| 100 (기본값: 20) |
| 정렬 | String | 선택 | "name" \| "reg_date" \| "status" (기본값: "reg_date DESC") |

#### 출력 항목

| 항목명 | 타입 | 설명 |
|--------|------|------|
| 회원 목록 | Array | [{id, name, phone, status, membership, manager, reg_date}, ...] |
| 총 건수 | Number | 필터 적용된 전체 회원 수 |
| 페이지정보 | Object | {current: 1, total: 5, limit: 20} |

#### 비즈니스 규칙
1. branch_id 기준 데이터 격리 (다지점 사용자는 자신의 지점만)
2. 권한별 조회 범위 제한:
   - 센터장: 전체 회원 조회
   - PT담당자: 자신이 담당한 회원만
   - 조회전용: 읽기만 가능
3. 삭제 회원 기본 미표시 (필터에서 "삭제 포함" 옵션으로 표시 가능)
4. 검색 debounce 300ms (API 호출 최소화)

#### 예외 처리

| 상황 | 대응 방법 |
|------|---------|
| 검색 결과 없음 | "검색 결과가 없습니다" 빈 상태 UI |
| 서버 오류 | "데이터를 불러올 수 없습니다. 새로고침을 시도해주세요" |
| 권한 없음 | "조회 권한이 없습니다" |

#### Web 구현 사항
- **테이블**: TanStack Table (React Table)
- **데이터 페칭**: React Query
- **검색 debounce**: useDebouncedValue (500ms)
- **엑셀 내보내기**: xlsx 라이브러리
- **필터 상태**: URL query parameters (뒤로가기 지원)

### FN-004: 회원 등록

| 항목 | 내용 |
|------|------|
| **기능 ID** | FN-004 |
| **기능명** | 회원 등록 |
| **분류** | 회원관리 |
| **플랫폼** | Web |
| **우선순위** | Critical |
| **상태** | 확정 |
| **화면 ID** | SCR-010 |
| **관련 API ID** | API-009 |

#### 기능 설명
신규 회원을 등록합니다. 필수 항목(이름, 성별, 연락처)은 반드시 입력해야 하며, 담당 FC를 배정할 수 있습니다. 동일 지점 내에서 연락처 중복을 방지합니다.

#### 상세 요구사항
1. 필수: 이름, 성별, 연락처
2. 선택: 이메일, 주소, 생년월일, 메모
3. 담당 FC 배정 (드롭다운)
4. 연락처 중복 체크 (API 호출)
5. 등록 완료 후 회원 상세 페이지로 자동 이동
6. 폼 유효성 검증 (실시간)

#### 입력 항목

| 항목명 | 타입 | 필수여부 | 설명 |
|--------|------|---------|------|
| 이름 | String | 필수 | 1~50자 |
| 성별 | String | 필수 | "M" \| "F" |
| 생년월일 | Date | 선택 | YYYY-MM-DD |
| 연락처 | String | 필수 | 전화번호 (01012345678 형식) |
| 이메일 | String | 선택 | 이메일 형식 검증 |
| 주소 | String | 선택 | 자유 텍스트 |
| 메모 | String | 선택 | 최대 500자 |
| 담당자ID | UUID | 선택 | FC 담당자 드롭다운 |

#### 출력 항목

| 항목명 | 타입 | 설명 |
|--------|------|------|
| 등록된 회원 정보 | Object | {id, name, phone, status, reg_date, ...} |

#### 비즈니스 규칙
1. 연락처 중복 불가 (동일 지점 내)
2. 등록 시 first_reg_date 자동 설정 (현재 일시)
3. 기본 상태: "활성"
4. 담당자 미배정 가능 (나중에 수정)

#### 예외 처리

| 상황 | 대응 방법 |
|------|---------|
| 중복 연락처 | "이미 등록된 연락처입니다. 기존 회원을 확인하시겠습니까?" + 기존 회원 링크 제시 |
| 필수 미입력 | 필드별 빨강 경고 + "필수 항목을 입력해주세요" |
| 연락처 형식 오류 | "올바른 전화번호 형식을 입력해주세요 (예: 010-1234-5678)" |
| 이메일 형식 오류 | "올바른 이메일 형식을 입력해주세요" |
| 서버 오류 | "회원 등록에 실패했습니다. 다시 시도해주세요" |

#### Web 구현 사항
- **Form 관리**: React Hook Form + Zod validation
- **폼 레이아웃**: 2열 그리드 (이름/성별 한 줄, 연락처 한 줄, 등)
- **담당자 선택**: Combobox (검색 가능)
- **연락처 중복 체크**: onChange 이벤트로 API 호출 (debounce 500ms)

### FN-005: 회원 정보 수정

| 항목 | 내용 |
|------|------|
| **기능 ID** | FN-005 |
| **기능명** | 회원 정보 수정 |
| **분류** | 회원관리 |
| **플랫폼** | Web |
| **우선순위** | High |
| **상태** | 확정 |
| **화면 ID** | SCR-011 |
| **관련 API ID** | API-010 |

#### 기능 설명
기존 회원의 정보를 수정합니다. 변경 이력을 자동으로 기록하며, 변경된 필드를 강조 표시합니다.

#### 상세 요구사항
1. 기존 데이터 프리필 (폼에 자동 입력)
2. 변경 필드 강조 (노란색 배경)
3. 변경 이력 자동 기록 (audit log)
4. 담당자 변경 가능
5. "저장" 전에 변경 내용 요약 미리보기
6. 취소 시 변경사항 폐기

#### 입력 항목
(FN-004와 동일한 항목들)

#### 출력 항목

| 항목명 | 타입 | 설명 |
|--------|------|------|
| 수정된 회원 정보 | Object | 변경된 모든 필드 포함 |
| 변경 이력 | Array | [{field, before, after, changed_at, changed_by}, ...] |

#### 비즈니스 규칙
1. 변경 이력 audit log 자동 생성
2. 연락처 변경 시 중복 체크
3. 저장 버튼 활성화: 최소 1개 필드 변경 시만
4. 삭제된 회원 수정 불가 (읽기전용)

#### 예외 처리

| 상황 | 대응 방법 |
|------|---------|
| 변경 없이 저장 | "변경된 내용이 없습니다" 안내 메시지 |
| 연락처 중복 | "이미 다른 회원이 사용 중인 연락처입니다" |
| 서버 오류 | "저장에 실패했습니다. 다시 시도해주세요" + 롤백 |
| 동시 수정 (충돌) | "다른 사용자에 의해 수정되었습니다. 새로고침 후 다시 시도해주세요" |

#### Web 구현 사항
- **Form 관리**: React Hook Form + 초기값 프리필
- **변경 감지**: useFormContext + watch로 변경된 필드 추적
- **변경 강조**: 변경된 필드에 CSS 클래스 적용
- **미리보기 모달**: 저장 전 변경 내용 요약 표시
- **비교 라이브러리**: diff-match-patch (변경 전/후 비교)

### FN-006: 회원 상세 조회

| 항목 | 내용 |
|-----|------|
| **기능 ID** | FN-006 |
| **기능명** | 회원 상세 조회 (탭별 정보) |
| **분류** | 회원관리 |
| **플랫폼** | Web |
| **우선순위** | High |
| **상태** | 확정 |
| **화면 ID** | SCR-011 |
| **관련 API ID** | API-008 |

#### 기능 설명
회원의 전체 정보를 탭 형식으로 조회합니다. 각 탭은 lazy loading으로 필요할 때만 데이터를 로드합니다.

#### 상세 요구사항
1. 프로필 탭: 기본 정보 + 사진
2. 이용권 탭: 보유 이용권 목록 + 상태
3. 출석 탭: 히트맵 + 월별/주별 리스트
4. 결제 탭: 결제 이력 + 영수증 다운로드
5. 체성분 탭: 측정 기록 + 차트
6. 메모 탭: 상담 메모/기록

#### 입력 항목

| 항목명 | 타입 | 필수여부 | 설명 |
|--------|------|---------|------|
| 회원ID | UUID | 필수 | URL 파라미터에서 추출 |
| 활성탭 | String | 선택 | "profile" \| "membership" \| "attendance" \| "payment" \| "body" \| "memo" |

#### 출력 항목

| 항목명 | 타입 | 설명 |
|--------|------|------|
| 회원 정보 | Object | 기본정보, 사진, 가입일 등 |
| 이용권 목록 | Array | [{id, type, start_date, end_date, status}, ...] |
| 출석 데이터 | Array | [{date, check_in_time, type}, ...] |
| 결제 이력 | Array | [{id, amount, method, date, receipt}, ...] |
| 체성분 기록 | Array | [{date, weight, body_fat, muscle, bmi}, ...] |
| 메모 목록 | Array | [{id, content, created_at, created_by}, ...] |

#### 비즈니스 규칙
1. 탭별 lazy loading (첫 로드 시 프로필만)
2. 이용권 만료일을 눈에 띄게 강조 (빨강)
3. 출석 히트맵: 최근 6개월
4. 각 탭 접근 권한 검증 (역할별)
5. 읽기 전용 (수정은 "회원 수정" 페이지에서)

#### 예외 처리

| 상황 | 대응 방법 |
|------|---------|
| 회원 미존재 | 404 페이지 또는 "존재하지 않는 회원입니다" |
| 조회 권한 없음 | "조회 권한이 없습니다" |
| 탭 데이터 로드 실패 | 해당 탭에 "데이터를 불러올 수 없습니다" 에러 표시 |

#### Web 구현 사항
- **탭 UI**: React Tabs 컴포넌트
- **Lazy Loading**: React Query의 suspense 또는 useQuery 동적 호출
- **출석 히트맵**: react-calendar 또는 커스텀 그리드
- **차트**: Recharts (체성분 변화)
- **병렬 로드**: 각 탭 쿼리 병렬 실행 (필요 시)

### FN-007: 체성분 기록/분석

| 항목 | 내용 |
|------|------|
| **기능 ID** | FN-007 |
| **기능명** | 체성분 기록/분석 |
| **분류** | 회원관리 |
| **플랫폼** | Web |
| **우선순위** | High |
| **상태** | 확정 |
| **화면 ID** | SCR-014 |
| **관련 API ID** | API-008 |

#### 기능 설명
회원의 체중, 골격근, 체지방률, BMI 등 체성분을 기록하고, 시계열 차트와 목표 대비 달성률을 분석합니다.

#### 상세 요구사항
1. 측정 항목: 체중, 골격근, 체지방률, BMI, 내장지방, 기초대사량
2. 측정일 기반 시계열 차트
3. 목표 설정 (목표 체중 등)
4. 목표 대비 달성률 게이지
5. 측정값 유효성 검증
6. 동일 날짜 중복 측정 시 덮어쓰기 확인

#### 입력 항목

| 항목명 | 타입 | 필수여부 | 설명 |
|--------|------|---------|------|
| 체중 | Number | 필수 | kg 단위 (20~300) |
| 골격근 | Number | 선택 | kg 단위 |
| 체지방률 | Number | 선택 | % 단위 (0~100) |
| BMI | Number | 선택 | 자동 계산 (체중/신장²) |
| 내장지방 | Number | 선택 | 레벨 (1~15) |
| 기초대사량 | Number | 선택 | kcal 단위 |
| 측정일 | Date | 필수 | YYYY-MM-DD |
| 메모 | String | 선택 | 측정 시 메모 |

#### 출력 항목

| 항목명 | 타입 | 설명 |
|--------|------|------|
| 체성분 이력 | Array | 측정 기록 리스트 |
| 변화 차트 데이터 | Array | [{date, weight, muscle, fat}, ...] |
| 달성률 | Number | (현재 체중 - 초기 체중) / (목표 - 초기) * 100 (%) |

#### 비즈니스 규칙
1. 측정값 유효 범위 검증:
   - 체중: 20~300kg
   - 체지방률: 0~100%
   - BMI: 수동 입력 시 10~60
2. 동일 날짜 중복 측정 시 "이 날짜의 측정 기록이 이미 존재합니다. 덮어쓰시겠습니까?" 확인
3. 목표 달성률 게이지: (현재값 - 초기값) / (목표값 - 초기값) * 100

#### 예외 처리

| 상황 | 대응 방법 |
|------|---------|
| 범위 초과 | "경고: 입력 범위를 벗어났습니다 (체중 20~300kg). 저장하시겠습니까?" (저장 허용) |
| 측정 기록 없음 | "측정 기록이 없습니다. 첫 기록을 입력해주세요" |
| 잘못된 형식 | 필드별 에러 메시지 표시 |

#### Web 구현 사항
- **차트**: Recharts (LineChart, AreaChart)
- **게이지**: Custom Gauge 또는 circular-progress 라이브러리
- **입력 폼**: React Hook Form + Number 입력
- **유효성**: Zod 또는 yup 스키마
- **날짜**: date-fns, dayjs

## 회원관리 공통 사항

### 데이터 격리
- 모든 회원 조회/수정은 branch_id 기준으로 현재 지점의 회원만 대상
- 다지점 관리자는 지점 전환 후 조회

### 권한 검증
| 기능 | 센터장 | PT담당자 | 직원 | 조회전용 |

- **차트**: Recharts
- **데이터 페칭**: React Query (TanStack Query)
- **상태**: Zustand (필터/정렬 상태)
