## 역할별 차이
- staff/readonly: A 상태에서 CTA 버튼 비표시 (안내만)
- trainer: 타이틀에 "본인 담당" 명시 — "본인 담당 일정 요청이 없습니다"
- super/primary: A 상태에서 BranchSwitcher 위치 유지(빈 지점 확인 가능)

## 상태 고유 인터랙션
- **필터 초기화 버튼** (B): `setFilter(DEFAULT_FILTER)` + URL 쿼리 제거 → refetch
- **안내 메시지 보내기** (A): phase 2 확장 — 현재는 `toast.info("준비 중입니다")`
- **RefreshButton**: 수동 refetch 가능 — 실시간 요청 도착 확인용
- **자동 폴링**: A 상태에 60초 이상 체류 시 silent refetch(새 요청 자동 감지, 결과 있으면 `02-정상`)

## 상태 고유 비즈니스 룰 델타
1. **A/B 판별 로직**:
   ```ts
   const isFiltered = typeFilter !== 'all' ||
     startDate !== DEFAULT_START || endDate !== DEFAULT_END;
   ```
2. **pendingCount 배너 유지(B)**: 필터 결과 없더라도 전체 pendingCount > 0 이면 배너는 유지. 사용자가 필터로 놓치는 상황 방지.
3. **자동 폴링**: A 상태 + 포커스 탭이면 60초 주기로 refetch. B 상태는 폴링하지 않음.
4. **빈 상태 진입 시간 기록**: analytics `event:schedule_requests_empty_view` 전송(A/B 구분).

## 에지 케이스
- **pending 있으나 과거 기간 필터로 0건**: B 상태. 배너는 "처리 대기 중 N건" 유지.
- **서버 timezone 이슈로 일부 데이터 누락**: KST 정오 근처 기간 경계 확인. 서버 `from`/`to`를 KST 기준 ISO 변환.
- **CSV 내보내기 후 0건 반환**: CSV 다운로드 대신 toast.warning "내보낼 요청이 없습니다"
- **신규 요청 수신(실시간)**: 서버 webhook + client polling → items 수신 → `02-정상`로 자연 전환.
