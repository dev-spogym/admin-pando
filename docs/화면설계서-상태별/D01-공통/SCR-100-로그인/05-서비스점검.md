## 상태 고유 인터랙션
1. **자동 상태 확인**: 30초마다 `GET /health` 재호출 → 200이면 `01-기본` 자동 복귀
2. **새로고침 버튼**: 즉시 상태 재확인 (`GET /health`) → 200이면 복귀
3. **문의 링크**: `mailto:support@fitgenie.kr` 또는 고객지원 페이지
4. **자동 복귀 시 토스트**: "서비스가 복구되었습니다" (우측 상단)

## 상태 고유 비즈니스 룰 델타
- **상태 폴링**: `useInterval(() => checkHealth(), 30_000)`
- **exponential backoff**: 3회 연속 실패 후 폴링 간격 60s → 120s → 300s 증가(최대 300s)
- **점검 종료 시간 표시**: 서버 응답 헤더 `Retry-After` 또는 `X-Maintenance-End` 파싱
- **예상 시간 미제공 시**: "잠시 후 다시 이용해주세요" 만 표시
- **Analytics 이벤트**: `maintenance_shown` 진입 시 1회 기록

## 에지 케이스
- **점검 종료 후 401/500 연속 발생**: 실제 서비스 문제 → `E500001`로 전환(03-에러)
- **네트워크 단절 + 점검 동시**: `NETWORK` 메시지 우선 표시 (점검 여부 확인 불가)
- **사용자가 오랫동안 이 화면에 체류**: 5분 후 폴링 중단 + "새로고침을 눌러 다시 확인해주세요"
- **Feature Flag 기반 점검**: 실제 `/health`는 200이지만 앱 레벨에서 차단 — 서버가 `maintenance_mode` 필드 반환하는지 확인

```
상태 폴링 로직:
const [estimatedEnd, setEstimatedEnd] = useState<Date | null>(null);
const [isChecking, setIsChecking] = useState(false);
const [pollInterval, setPollInterval] = useState(30_000);
const failCount = useRef(0);

async function checkHealth() {
  setIsChecking(true);
  try {
    const res = await fetch('/api/health', { cache: 'no-store' });
    if (res.ok) {
      toast.success('서비스가 복구되었습니다');
      setStep('credentials'); // 01-기본 복귀
      return;
    }
    const maintEnd = res.headers.get('X-Maintenance-End');
    if (maintEnd) setEstimatedEnd(new Date(maintEnd));
    failCount.current++;
    setPollInterval(Math.min(30_000 * 2**failCount.current, 300_000));
  } catch { failCount.current++; }
  finally { setIsChecking(false); }
}

useEffect(() => {
  const id = setInterval(checkHealth, pollInterval);
  return () => clearInterval(id);
}, [pollInterval]);

- 새로고침 버튼 aria-busy={isChecking}
- H1 id="maintenance-title", 카드에 aria-labelledby="maintenance-title"
```
