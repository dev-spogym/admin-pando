## 상태 진입 조건
- 부모 화면의 삭제 버튼 클릭
- 클라이언트 `canDelete(role, resource)` 체크 통과
- `useSessionStore.open === false` (세션만료 우선)

## 상태 고유 비주얼 델타
- 기본 렌더(마스터 §2)
- "취소" 버튼 `autoFocus`
- 타이핑 확인(`confirmationText`) 옵션이 있으면 입력 필드 노출
- `loading=false`

## 상태 고유 인터랙션
1. **오픈 직후**: "취소" 포커스(안전 기본값). Enter 누르면 취소가 기본 동작
2. **타이핑 입력**: 매칭 시 Primary 버튼 활성
3. **ESC / 배경 / X**: 취소와 동일
4. **"삭제"**: 검증 통과 시 `02-삭제중` 전환
5. **Tab 순환**: 취소 → 입력 필드 → 삭제 → X → 취소

## 역할별 차이
- 부모가 권한 없다고 판단하면 다이얼로그 자체가 오픈되지 않음
- `primary/super`는 지점 간 데이터 삭제 가능 → branchId 스코프 명시 UI (제목에 지점명 포함 권장)

## 상태 고유 비즈니스 룰 델타
- 일괄 삭제의 경우 `confirmLabel = "${N}건 삭제"` 로 치환
- 참조 카운트가 `>= N (예: 100)` 인 대규모 삭제는 타이핑 확인 강제(`confirmationText='삭제'`)
- 다이얼로그 열린 채 다른 행의 삭제 버튼 누르면: 기존 다이얼로그 닫고 새 다이얼로그 오픈(target 교체)

## 에지 케이스
- **이미 삭제된 항목**: 오픈은 정상이지만 확정 시 404 → `04-삭제실패` 에서 "이미 삭제된 항목입니다" 토스트 후 목록 refetch
- **권한 중간 변경**: 오픈 후 역할이 변경되어 삭제 권한 박탈 → 서버 403 응답으로 방어
- **네트워크 단절 중 확정**: `02-삭제중` 에서 대기 → 네트워크 에러 → 다이얼로그 유지
- **i18n**: 본문의 `"${name}"을(를)` 처리 시 한국어 조사 처리는 `isKoreanVowel` 유틸 활용 권장
