## 역할별 차이
- 없음 (제출 중 공통 UX)

## 상태 고유 인터랙션
1. **모든 입력 잠금** (Radio, textarea, 취소, X, Esc, Overlay 클릭)
2. **중단 불가**: 네트워크 요청은 브라우저 `abortController`로도 UI 버튼 미제공 (API 멱등 아님 우려)
3. **타임아웃**: 30초 경과 시 자동 `04-성공또는실패` (네트워크 타임아웃 처리)

## 상태 고유 비즈니스 룰 델타
1. 제출 중 폼 자체 편집 차단
2. 제출 중 뒤로가기/닫기 모두 차단 (Radix + `beforeunload` 경고 옵션)
3. 낙관적 UI 금지 (일괄 상태 전이는 서버 결과 확정 후 UI 반영)
4. 응답 30초 초과 → `E500001` 토스트 + `04-성공또는실패` 전환 (isSubmitting=false)

## 에지 케이스
- **사용자가 창 닫기 시도**: `beforeunload` 이벤트로 "작업이 진행 중입니다. 정말 나가시겠습니까?" 경고
- **네트워크 단절**: `mutation.onError` → `NETWORK` 코드 → `04-성공또는실패` 에러 상태
- **서버 500**: 재시도 가능한 상태로 복귀 (`04` 에러 서브상태)
- **동시에 다른 사용자가 동일 회원 수정**: 서버 낙관적 락 → E409101

## 바이브코딩 프롬프트 (델타)
```
상태: submitting
조건: isSubmitting === true (mutation.isPending)
UI 변경:
  - Dialog.Close disabled / aria-disabled
  - RadioGroup className: pointer-events-none opacity-70
  - textarea readOnly
  - 취소/확인 버튼 disabled
  - 확인 버튼 children: <><Loader2 className="animate-spin" size={14}/>변경 중...</>
  - onEscapeKeyDown: e.preventDefault()
  - onPointerDownOutside: e.preventDefault()
beforeunload:
  useEffect(() => {
    if (!isSubmitting) return;
    const h = (e: BeforeUnloadEvent) => { e.preventDefault(); e.returnValue = ''; };
    window.addEventListener('beforeunload', h);
    return () => window.removeEventListener('beforeunload', h);
  }, [isSubmitting]);
타임아웃:
  AbortSignal.timeout(30_000) 적용 (fetch 기반 시)
```
