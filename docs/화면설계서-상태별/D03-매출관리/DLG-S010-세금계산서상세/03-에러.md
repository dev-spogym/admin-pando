## 4. 역할별 차이

- fc 역할은 RLS 위반 케이스에서 가장 흔하게 만남(지점 범위 외 조회 시도 등). 메시지는 동일.
- 본질적 차이는 없으며 모든 역할에서 토스트+닫힘 동일.

## 5. 상태 고유 인터랙션

| 이벤트 | 트리거 | 결과 |
|---|---|---|
| `E_LOAD_ERR_01` | fetch 실패 | toast + onClose |
| `E_RETRY` | 사용자가 목록에서 다시 행 클릭 | `01-로딩` 재진입(재시도) |
| `E_DISMISS_TOAST` | 사용자가 토스트 X 클릭 | 토스트 즉시 제거 |

> 본 상태는 "유지형 에러"가 아니라 **즉시 통지 후 닫힘**형. 재시도는 SCR-S010에서 수행.

---

## 6. 상태 고유 비즈니스 룰 델타

1. **즉시 닫힘**: 조회 전용이므로 에러 시 모달을 유지할 필요 없음. 사용자는 목록에서 재시도.
2. **에러 코드 매핑**:
   - `PGRST116` → "존재하지 않는 세금계산서입니다."
   - `42501` → "조회 권한이 없습니다."
   - `Failed to fetch` → "네트워크 오류. 연결을 확인해주세요."
   - 기타 → "데이터를 불러올 수 없습니다."
3. **토스트 단일화**: id `'tax-invoice-detail-error'` 로 중복 방지.
4. **부모 목록 무효화(옵션)**: 404 에러는 목록에서 해당 항목 제거 가능(`invalidateQueries`).
5. **감사로그**: 조회 에러는 서버에 기록하지 않음(비민감).

## 7. 에지 케이스

| 케이스 | 처리 |
|---|---|
| 로딩 중 이미 사용자가 ESC로 닫음 | 에러 콜백에서 `mounted=false` 체크 → 토스트 생략 가능 |
| 네트워크 단절 후 즉시 복귀 | 사용자가 행 재클릭 시 정상 로딩 |
| RLS 위반으로 404 발생 | 메시지는 "존재하지 않음"으로 통일(403과 구분 안 함) — 보안상 정책 |
| 세션 만료(401) | DLG-000 우선, 본 에러 토스트는 억제 |
| 동일 에러 반복 | 토스트 id 재사용으로 중복 스택 방지 |

## 8. 바이브코딩 프롬프트 (델타)

```
상태: DLG-S010 — 03-에러

// fetchTaxInvoiceDetail 내부 에러 분기
const { data, error } = await supabase
  .from('tax_invoices')
  .select('*')
  .eq('id', taxInvoiceId)
  .single();

if (!mounted) return;                   // 01-로딩 취소 케이스
if (error) {
  toast.error(mapInvoiceError(error), { id: 'tax-invoice-detail-error' });
  setIsLoading(false);
  onClose();                            // 즉시 모달 닫힘
  return;
}

function mapInvoiceError(err: any): string {
  if (err?.code === 'PGRST116') return '존재하지 않는 세금계산서입니다.';
  if (err?.code === '42501') return '조회 권한이 없습니다.';
  if (err?.message?.includes('Failed to fetch')) return '네트워크 오류. 연결을 확인해주세요.';
  if (err?.code === 'PGRST301') return '세션이 만료되었습니다.';
  return '데이터를 불러올 수 없습니다.';
}
```
