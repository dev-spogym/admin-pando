# 전사 정합성 정리안

> 목적: 현재 문서군에 분산된 용어, 상태, 역할, 자동화 기준을 퍼블리싱 가능한 하나의 기획 기준으로 재정리한다.

---

## 1. 정리 대상

현재 충돌이 발생한 핵심 축은 아래 5개다.

1. 리드 파이프라인 상태
2. 회원/이용권/결제 상태
3. 관리자 웹 권한 체계
4. 회원앱 역할 체계
5. 자동화/만료 알림 정책

---

## 2. 상위 원칙

### 2.1 관리자 웹과 회원앱은 같은 데이터 원장을 보지만 같은 역할 체계를 쓰지 않는다

- 관리자 웹은 운영용 Backoffice다.
- 회원앱은 회원 중심 모바일 채널이다.
- 두 채널은 데이터를 공유하되, 화면 노출 기준과 역할 체계는 분리한다.

### 2.2 상태값과 표시문구를 분리한다

- DB 저장 상태와 UI 표시 상태를 혼용하지 않는다.
- 예: `INACTIVE`와 `미등록`은 동일 개념이 아니다.
- 예: `만료 임박`은 DB 저장 상태가 아니라 UI 계산 상태다.

### 2.3 자동화는 지점별 자유 생성보다 본사 정책 라이브러리 기반 선택형이 우선이다

- 만료 알림은 센터마다 조금씩 다를 수 있으나, 운영 품질은 본사가 통제해야 한다.
- 따라서 "본사 등록 -> 지점 선택 적용" 구조를 기본으로 한다.

### 2.4 퍼블리싱 기준은 백엔드 미구현 여부와 분리해 확정한다

- 실제 연동 구현 여부와 무관하게 화면의 상태, 문구, 버튼 활성/비활성 규칙은 먼저 고정한다.
- 미구현은 별도 비고로 남기되 상태 체계를 흔들지 않는다.

---

## 3. 도메인 간 유기적 흐름 기준

아래 흐름이 이 프로젝트의 핵심 운영 체인이다.

1. 리드 유입
2. 상담/후속 조치
3. 회원 전환
4. 결제 승인
5. 이용권 생성/활성화
6. 예약/수업/출석
7. 만족도/후기/체성분/상담 누적
8. 만료 임박 알림
9. 재등록 유도
10. 환불/정지/휴면/탈퇴 처리

위 체인을 기준으로 보면, 현재 문서에서 반드시 먼저 고정해야 하는 것은 다음이다.

- 리드 상태가 어느 시점에 회원 전환으로 이어지는가
- 결제 완료가 어느 시점에 이용권 활성화로 이어지는가
- 이용권 만료 임박/만료/재등록이 어떤 배너와 알림으로 이어지는가
- 권한별로 누가 조회만 가능하고 누가 상태를 바꿀 수 있는가

## 4. 이번 정리안의 확정 기준

### 4.1 리드

- 리드는 "영업 파이프라인"으로 해석한다.
- 상태 표기는 운영자 친화적으로 한글을 사용하되, 내부 기준 키는 별도로 고정한다.
- 전환 완료 시 `convertedMemberId` 연결은 설계상 필수로 본다.
- 단, 연결 실패 상태는 "등록완료이지만 미연결" 경고 상태로 노출 가능하다.

### 4.2 회원

- 회원은 DB 저장 상태와 화면 계산 상태를 분리한다.
- `WITHDRAWN`, `DORMANT`, `TRANSFERRED`를 `INACTIVE`에 흡수하지 않는다.
- `미등록`은 저장 상태가 아니라 표시 상태다.

### 4.3 이용권

- 이용권은 회원보다 더 직접적인 운영 기준이다.
- 결제/환불/홀딩/만료는 이용권 상태를 우선 기준으로 본다.
- 회원 대표 상태는 보유 이용권 상태를 바탕으로 계산한다.

### 4.4 결제

- 결제 상태는 "정상 승인", "미수/분할 진행", "부분환불", "전액환불", "실패", "취소"로 분리한다.
- 기존 `COMPLETED/UNPAID` 기반 화면은 새 상태 체계에 매핑해 유지 가능하다.

### 4.5 자동화

- 만료 알림은 더 이상 고정 `D-30/D-14/D-7/D-3/D-1` 전제 문장으로만 쓰지 않는다.
- 본사가 규칙 세트를 만들고, 지점은 세트 또는 세트 일부를 선택한다.

## 5. 문서 간 충돌 시 해석 원칙

### 5.1 리드 상태

- 최종 기준: `02_상태체계_통합안.md`
- 기존 문서의 `INQUIRY`, `PENDING`, `상담예정`, `방문완료`, `HOLD`는 모두 통합 매핑표로 흡수한다.

### 5.2 회원 상태

- 최종 기준: `02_상태체계_통합안.md`
- `INACTIVE = 미등록` 해석은 폐기한다.

### 5.3 권한

- 최종 기준: `03_권한체계_통합안.md`
- 관리자 웹에서 `trainer`, `front`는 시스템 표준 역할로 보지 않는다.

### 5.4 만료 알림

- 최종 기준: `04_자동화_만료알림_정책_개정안.md`
- 지점은 본사 제공 정책만 선택 적용한다.

---

## 6. 퍼블리싱 관점 핵심 지시

퍼블리싱 시 아래 기준을 우선 적용한다.

- 상태 배지 문구는 통합안 기준으로만 쓴다.
- 버튼 활성/비활성은 권한체계 통합안 기준으로만 쓴다.
- "미구현"인 기능도 상태/레이아웃/CTA는 통합안 기준으로 고정한다.
- 같은 개념에 서로 다른 한글명이 섞이면 통합안 한글명을 우선한다.
- 회원앱 공통 화면에 "별도 CRM 앱 사용" 같은 문구는 통합 모바일 앱 전략과 충돌하므로 제거 대상이다.

---

## 7. 최종 목표

이 정리안의 목적은 "문서가 많다"가 아니라 "누가 봐도 같은 화면을 같은 기준으로 퍼블리싱할 수 있다"는 상태를 만드는 것이다.

따라서 이후 정비는 아래 순서를 따른다.

1. 상태 체계 확정
2. 권한 체계 확정
3. 자동화/만료 알림 정책 확정
4. 회원앱 포지션 문구 정리
5. 각 화면 마스터 문서 반영
