## 1. 도메인 개요

직원관리 도메인은 피트니스 센터(지점)에 소속된 직원의 **라이프사이클 전체**를 관리한다.

```
등록(ACTIVE) ──→ 역할변경 ──→ 휴직(ON_LEAVE) ──→ 복직(ACTIVE)
     │                              │
     │                              ▼
     └──→ 퇴사예정 ──→ 퇴사확정(RESIGNED)
     │
     └──→ 전보(TRANSFERRED) ──→ 타지점 ACTIVE
```

**하위 기능 영역**:

| 영역 | 라우트 | 설명 |
|------|--------|------|
| 직원 목록 | `/staff` | 전체 직원 조회, 필터, 검색, 일괄 퇴사 |
| 직원 등록 | `/staff/new` | 신규 직원 등록 폼 |
| 직원 수정 | `/staff/edit?id={id}` | 기존 직원 정보 수정 (등록과 동일 폼) |
| 퇴사 처리 | `/staff/resignation` | 4단계 위자드 기반 퇴사 프로세스 |
| 직원 근태 | `/staff/attendance` | 일별/월별 출퇴근 기록 및 근태 현황 |
| 급여 관리 | `/payroll` | 월별 급여 조회, 확정, 내보내기 |
| 급여 명세서 | `/payroll/statements` | 직원별 명세서 상세 조회 및 발급 |

### 4.6 퇴사 처리 (간이 -- 목록에서의 일괄 처리)

**ConfirmDialog 설정**:

| 속성 | 값 |
|------|-----|
| title | "직원 퇴사 처리" |
| description | "선택한 {n}명의 직원을 퇴사 처리하시겠습니까?\n퇴사 처리 시 해당 직원의 계정 접근이 제한됩니다." |
| confirmLabel | "퇴사 처리" |
| variant | danger |
| confirmationText | "퇴사처리" (사용자가 직접 타이핑해야 확인 가능) |

**처리 로직**:
1. 선택된 직원 ID 배열 추출 (`Array.from(selectedRows)`)
2. Supabase: `staff` 테이블에서 해당 ID들의 `isActive`를 `false`로 업데이트
3. 성공 시 토스트: "{n}명의 퇴사 처리가 완료되었습니다."
4. 다이얼로그 닫기 + 선택 초기화 + 목록 재조회 (`fetchStaff()`)

### 4.7 엑셀 내보내기

**내보내기 컬럼**: No, 직원명, 역할, 연락처, 입사일, 상태, 메모
**파일명**: `직원목록.xlsx`
**대상 데이터**: 현재 필터 조건이 적용된 전체 데이터 (페이지네이션 무관)

### 4.8 데이터 로드

- **소스**: Supabase `staff` 테이블 직접 쿼리
- **조건**: `branchId` = 현재 지점 ID (`localStorage.getItem('branchId')`)
- **컬럼**: `id, name, phone, email, role, position, hireDate, salary, color, isActive, branchId`
- **상태 매핑**: `isActive === false` -> `"resigned"`, 그 외 -> `"active"`
- **페이지네이션**: 클라이언트 사이드, 10건/페이지

## 6. SCR-062 퇴사 처리 (4단계 위자드)

**라우트**: `/staff/resignation`
**소스**: `src/app/staff/resignation/page.tsx`
**참조**: UC-025

### 6.1 위자드 개요

퇴사 처리는 4단계 위자드(Step Indicator)를 통해 진행된다. FC 역할이 아닌 직원은 Step 2(담당 회원 재배정)를 건너뛴다.

```
[1. 직원/퇴사 정보] ──→ [2. 담당 회원 재배정*] ──→ [3. 스케줄 확인] ──→ [4. 최종 확인]
                          (* FC 역할만)
```

**Step Indicator UI**: 원형 번호(1~4) + 연결선, 완료 단계는 체크 아이콘, 현재 단계는 링 효과

### 6.2 Step 1: 직원/퇴사 정보

**목적**: 퇴사 대상 직원 선택 및 퇴사 기본 정보 입력

#### 6.2.1 직원 선택

| 필드 | 필수 | 타입 | 동작 |
|------|:---:|------|------|
| 직원 | O | Select 드롭다운 | 활성 직원 목록에서 선택 (API: `getStaff({ size: 100 })`) |

- URL 쿼리 파라미터 `?staffId={id}`로 직원 사전 선택 가능
- 직원 선택 시 상세 정보 카드 표시 (이름, 역할 배지, 입사일, 담당 회원 수)
- 상세 로드: `getStaffById(selectedStaffId)`

#### 6.2.2 퇴사 정보

| 필드 | 필수 | 타입 | 유효성 |
|------|:---:|------|--------|
| 퇴사 예정일 | O | date input | `min=오늘`, 빈 값 불가 |
| 퇴사 사유 | - | textarea | 자유 텍스트 (placeholder: "퇴사 사유를 입력하세요 (개인 사정, 계약 만료 등)") |

#### 6.2.3 다음 단계 분기 로직

```
Step 1 -> [다음] 클릭
  ├─ 유효성 검증 실패 -> 에러 메시지 표시
  ├─ FC 역할 -> Step 2 (담당 회원 재배정)
  └─ FC 외 역할 -> Step 3 (스케줄 확인, Step 2 스킵)
```

### 6.3 Step 2: 담당 회원 재배정 (FC 역할만)

**목적**: FC 직원이 퇴사할 때 담당 회원을 다른 FC에게 재배정

#### 6.3.1 데이터 로드 (Step 2 진입 시)

두 API를 병렬 호출:
1. `getStaffMembers(staffId)` -> 퇴사 대상 FC의 담당 회원 목록
2. `getStaff({ size: 100, role: 'FC' })` -> 활성 FC 목록 (퇴사 대상 제외)

#### 6.3.2 일괄 배정

| 구성 요소 | 동작 |
|-----------|------|
| FC 드롭다운 | 활성 FC 목록에서 선택 |
| [미배정 회원에 적용] 버튼 | `assignedStaffId === null`인 모든 회원에 선택된 FC 일괄 적용 |

#### 6.3.3 회원 테이블

| 컬럼 | 렌더링 |
|------|--------|
| 회원명 | 텍스트 (font-medium) |
| 연락처 | 텍스트 (content-secondary) |
| PT 잔여 | "PT잔여" 빨간 배지 (잔여 있음) 또는 "-" (없음) |
| 재배정 담당자 | Select 드롭다운 (개별 FC 선택). PT 잔여 회원이 미배정이면 에러 표시 |

#### 6.3.4 다음 단계 진행 조건

- PT 잔여 회원 중 미배정 건수 > 0 -> 경고 토스트: "PT 잔여 회원 {n}명의 재배정 담당자를 지정해주세요."
- PT 잔여 없는 회원은 미배정 상태로 진행 가능 (자동 균등배분 대상)
- 상단 경고 배너: "PT 잔여 {n}명 미배정" (warning 스타일)

### 6.4 Step 3: 미래 스케줄 확인

**목적**: 퇴사 예정일 이후 잡혀 있는 스케줄 확인 및 처리 방법 결정

#### 6.4.1 스케줄 없는 경우

- 체크 아이콘(초록) + "퇴사 예정일 이후 예정된 스케줄이 없습니다." 메시지

#### 6.4.2 스케줄 있는 경우

**스케줄 테이블**:

| 컬럼 | 렌더링 |
|------|--------|
| 스케줄명 | 텍스트 |
| 유형 | PT(파란 배지) 또는 GX(초록 배지) |
| 날짜 | 텍스트 |

**처리 방법 선택** (RadioGroup, 세로 배치):

| 옵션 | 설명 |
|------|------|
| 후임에게 일괄 이관 | 재배정된 담당자에게 스케줄을 이관합니다. |
| 일괄 취소 | 모든 스케줄을 취소합니다. |

### 6.5 Step 4: 최종 확인

**목적**: 전체 퇴사 처리 내용을 요약하고 최종 확인

#### 6.5.1 경고 배너

- AlertTriangle 아이콘 + "퇴사 예정 등록 후에는 수정이 어려울 수 있습니다. 아래 내용을 확인 후 진행하세요."
- error 스타일 배경

#### 6.5.2 요약 카드 항목

| 항목 | 값 |
|------|-----|
| 퇴사 대상 | `{이름} ({역할})` |
| 퇴사 예정일 | `YYYY년 M월 D일` (한국어 날짜 포맷) |
| 퇴사 사유 | 입력값 또는 "(미입력)" |
| 담당 회원 재배정 | `{n}명 완료` (FC인 경우에만 표시) |
| 미래 스케줄 | "예정된 스케줄 없음" 또는 "{n}건 후임 이관" / "{n}건 일괄 취소" |

#### 6.5.3 최종 제출 처리 흐름

```
[퇴사 예정 등록] 클릭
  │
  ├─ (FC이고 배정 내역이 있을 때)
  │   1. reassignMembers({ assignments }) 호출
  │      - assignedStaffId가 null이 아닌 회원들만 추출
  │      - 각 회원의 staffId를 새 담당자로 업데이트
  │      - 실패 시 에러 토스트 + 중단
  │
  ├─ 2. scheduleResignation(staffId, { resignScheduledAt, resignReason }) 호출
  │      - staff 테이블에 resignScheduledAt, resignReason 업데이트
  │      - staffStatus는 ACTIVE 유지 (예정일이므로)
  │      - 실패 시 에러 토스트 + 중단
  │
  └─ 3. 성공 토스트: "퇴사 예정이 등록되었습니다."
       → 직원 목록 화면으로 이동
```

### 6.6 퇴사 확정 (별도 프로세스)

퇴사 예정일이 도래하면 별도로 `confirmResignation(staffId)` API를 호출하여 확정 처리:
- `staffStatus` -> `"RESIGNED"`
- `resignedAt` -> 현재 시각
- `isActive` -> `false`

|-----------|-----------|

| `clockIn` 있고 `clockOut` 없음 | [퇴근] 버튼 (프라이머리) | `clockOut()` API 호출 -> 현재 시각을 `clock_out`에 기록, `work_hours` 계산 |
| 둘 다 있음 | "완료" 텍스트 | 추가 액션 없음 |

**근무시간 계산**: `(clock_out - clock_in) / 3600000` (밀리초 -> 시간, 소수점 2자리)

### 7.7 월별 요약 뷰

**데이터 소스**: `staff_attendance` 테이블에서 선택 월의 전체 기간(1일~말일) 조회
**지점 필터**: `branchId` = 현재 지점

| 컬럼 | key | 너비 | 렌더링 |
|------|-----|------|--------|

| 직급 | `role` | 100px | 텍스트 |

| 결근횟수 | `absentCnt` | 100px | `{n}회` (0이면 회색, 1 이상이면 빨간색) |
| 연차사용 | `annualCnt` | 100px | `{n}일` (0이면 회색, 1 이상이면 파란색) |

**동작**: 전 직원의 미지급/보류 급여를 일괄 "paid"로 변경

1. `window.confirm("전 직원 급여를 확정하시겠습니까?")` -> 취소 가능
2. `status !== "paid"`인 레코드 필터링

4. Supabase: `payroll.update({ status: "paid" }).in("id", ids)`
5. 성공 토스트: "급여가 확정되었습니다"
6. 클라이언트 상태 즉시 업데이트 (재조회 없이)

## 11. 비즈니스 규칙 요약

### 11.1 직원 라이프사이클

| 규칙 ID | 규칙 | 상세 |
|---------|------|------|
| BR-STAFF-001 | 직원 상태 전이 | ACTIVE -> ON_LEAVE, ACTIVE -> RESIGNED, ACTIVE -> TRANSFERRED만 허용 |
| BR-STAFF-002 | 퇴사 예정 등록 | `resignScheduledAt` 설정 시 `staffStatus`는 ACTIVE 유지 (예정일까지 근무) |
| BR-STAFF-003 | 퇴사 확정 | `confirmResignation` 호출 시 `isActive=false`, `staffStatus=RESIGNED`, `resignedAt=now()` |
| BR-STAFF-004 | FC 퇴사 시 회원 재배정 | FC 역할 직원 퇴사 시 담당 회원 재배정 필수 (PT 잔여 회원은 반드시 신규 담당자 지정) |
| BR-STAFF-005 | 휴직 처리 | `startStaffLeave` 호출 시 `isActive=false`, `staffStatus=ON_LEAVE` |
| BR-STAFF-006 | 복직 처리 | `endStaffLeave` 호출 시 `isActive=true`, `staffStatus=ACTIVE` (휴직 이력 유지) |
| BR-STAFF-007 | 전보 처리 | 원 지점 레코드 `TRANSFERRED`+비활성화, 신 지점에 새 레코드 생성 (`previousEmployeeId`로 연결) |
| BR-STAFF-008 | 역할 매핑 | DB 저장은 한글 (`센터장`), 폼 내부는 영문 키 (`owner`). 저장/로드 시 양방향 변환 |

### 11.2 근태 규칙

| 규칙 ID | 규칙 | 상세 |
|---------|------|------|
| BR-ATT-001 | 출근 기록 | 출근 버튼 클릭 시 현재 시각(`new Date().toISOString()`)을 `clock_in`에 기록, `status=정상` |
| BR-ATT-002 | 퇴근 기록 | 퇴근 버튼 클릭 시 현재 시각을 `clock_out`에 기록, `work_hours = (clock_out - clock_in) / 3600000` |
| BR-ATT-003 | 근태 미기록 직원 | 해당일 근태 기록이 없는 직원은 목록에서 `status=결근`, `clockIn/clockOut=null`로 표시 |
| BR-ATT-004 | 월별 요약 계산 | 출근일수 = 정상+지각+조퇴, 지각/결근/연차는 각 상태 개별 카운트 |
| BR-ATT-005 | 근태 상태 종류 | 정상, 지각, 조퇴, 결근, 연차, 휴무 (6종) |

### 11.3 급여 규칙
