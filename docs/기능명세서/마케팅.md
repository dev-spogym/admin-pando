## 1. 리드 관리

> **라우트**: `/leads`
> **소스**: `src/app/(marketing)/leads/page.tsx`
> **API**: `src/api/endpoints/leads.ts`
> **유저케이스**: UC-026

### 1.1 화면 개요

아직 회원이 아닌 잠재 고객(문의/상담)을 등록하고, 파이프라인 형태로 전환 과정을 추적하는 화면이다. **목록 뷰**와 **칸반 뷰** 두 가지 모드를 지원한다.

### 1.2 상단 통계 카드 (5열)

| 카드 | 값 | 아이콘 | 계산 로직 |
|------|-----|--------|-----------|
| 전체 리드 | `{total}건` | Users | 전체 리드 수 |
| 등록 전환 | `{converted}건` | UserPlus | status === '등록완료' 건수 |
| 전환율 | `{conversionRate}%` | TrendingUp | (등록완료 / 전체) * 100 |
| 대기 중 | `{pending}건` | Phone | status === '신규' + '연락완료' 건수 |
| 미전환율 | `{missedRate}%` | AlertCircle | (미전환 / 전체) * 100 |

### 1.3 상태별 분포 섹션

전체 리드가 1건 이상일 때 표시. 각 상태별 건수와 비율(%)을 그리드 카드로 표시한다. 건수가 0인 상태는 숨긴다.

### 1.4 필터 및 뷰 전환

| 필터 | 유형 | 선택지 |
|------|------|--------|
| 상태 | Select | 전체 상태 / 신규 / 연락완료 / 상담예정 / 방문완료 / 등록완료 / 미전환 / 보류 |
| 유입경로 | Select | 전체 유입경로 / 간판 / 인터넷 / 전단지 / 추천 / SNS / 카카오톡 / 전화문의 / 방문 / 기타 |

**뷰 전환 토글**: 목록 / 칸반 (버튼 그룹)

### 1.5 목록 뷰 (DataTable)

| 컬럼 | key | width | 렌더링 |
|------|-----|-------|--------|
| 문의일 | inquiryDate | 100 | YYYY-MM-DD, 모노스페이스 |
| 이름 | name | - | 볼드 |
| 연락처 | phone | - | 모노스페이스 |
| 유입경로 | source | - | StatusBadge(info) |
| 상태 | status | center | StatusBadge(dot) + 상태별 variant |
| 담당 FC | assignedFc | - | - |
| 후속일 | followUpDate | 100 | YYYY-MM-DD |
| 메모 | memo | - | truncate 160px |
| 관리 | actions | center | 수정(Edit) + 삭제(Trash2) 아이콘 버튼 |

**검색**: 이름, 연락처, 담당자 텍스트 검색 (클라이언트 필터링)

### 1.6 칸반 뷰 (5열 파이프라인)

| 단계 | 매핑 status | 컬러 | 헤더 배경 |
|------|-------------|------|-----------|
| 신규 | 신규 | blue | bg-blue-50 |
| 상담예약 | 상담예정 | amber | bg-amber-50 |
| 상담완료 | 방문완료 | purple | bg-purple-50 |
| 등록의향 | 연락완료 | orange | bg-orange-50 |
| 등록완료 | 등록완료 | green | bg-green-50 |

각 열에는 해당 상태의 리드 카드가 표시된다. 카드 클릭 시 수정 모달이 열린다.

**칸반 카드 구성**:
- 이름 (13px 볼드)
- 연락처 (11px 모노스페이스)
- 유입경로 배지 + 메모 일부 (10px truncate)

### 1.7 리드 등록/수정 모달

| 필드 | 타입 | 필수 | 기본값 | 비고 |
|------|------|:---:|--------|------|
| 이름 | Input(text) | * | - | 빈 값 시 "이름을 입력하세요" 토스트 |
| 연락처 | Input(tel) | - | - | 010-0000-0000 포맷 |
| 유입경로 | Select | * | 전화문의 | 9종 선택지 |
| 상태 | 버튼 그룹 | - | 신규 | 7종 pill 버튼 |
| 담당 FC | Input(text) | - | - | 담당자 이름 직접 입력 |
| 문의일 | Input(date) | - | 오늘 | - |
| 후속 예정일 | Input(date) | - | null | 비워두면 null |
| 메모 | Textarea(3행) | - | - | 문의 내용, 특이사항 |

**저장 로직**:
- 신규: `createLead(branchId, ...form)` → 성공 토스트 → 목록 새로고침
- 수정: `updateLead(id, ...form)` → 성공 토스트 → 목록 새로고침
- 실패: 오류 토스트 표시

### 1.8 리드 삭제

ConfirmDialog: "이 리드를 삭제하시겠습니까?" → [삭제] 확인 → `deleteLead(id)` → 성공 토스트

### 1.9 리드 상태 배지 variant 매핑

| 상태 | variant |
|------|---------|
| 신규 | info |
| 연락완료 | default |
| 상담예정 | warning |
| 방문완료 | info |
| 등록완료 | success |
| 미전환 | error |
| 보류 | default |

### 1.10 API 엔드포인트

| 함수 | 테이블 | 동작 | 파라미터 |
|------|--------|------|----------|
| `getLeads(branchId)` | leads | SELECT * WHERE branchId, ORDER BY createdAt DESC | branchId |
| `createLead(input)` | leads | INSERT → RETURNING | CreateLeadInput |
| `updateLead(id, input)` | leads | UPDATE WHERE id | id, Partial<CreateLeadInput> |
| `deleteLead(id)` | leads | DELETE WHERE id | id |
| `getLeadStats(branchId)` | leads | 클라이언트 집계 | branchId |

### 1.11 데이터 타입

```typescript
type LeadSource = '간판' | '인터넷' | '전단지' | '추천' | 'SNS' | '카카오톡' | '전화문의' | '방문' | '기타';
type LeadStatus = '신규' | '연락완료' | '상담예정' | '방문완료' | '등록완료' | '미전환' | '보류';

interface Lead {
  id: number;
  branchId: number;

  phone: string | null;
  source: LeadSource;
  status: LeadStatus;
  assignedFc: string | null;
  memo: string | null;
  inquiryDate: string;
  followUpDate: string | null;
  convertedMemberId: number | null;
  createdAt: string;
}
```

## 2. 메시지 발송

> **라우트**: `/message`
> **소스**: `src/app/(marketing)/message/page.tsx`
> **유저케이스**: UC-027

### 2.1 화면 개요

회원에게 알림톡, SMS/LMS, 앱 푸시를 발송하고, 발송 이력을 조회하는 화면이다. 상단 탭으로 **메시지 전송** / **발송 이력** 두 화면을 전환한다.

### 2.2 상단 탭

| 탭 key | 라벨 | 아이콘 |
|--------|------|--------|
| send | 메시지 전송 | Send |
| history | 발송 이력 | History |

### 2.3 발송 채널 설정

| 채널 key | 라벨 | 최대 글자수 | 건당 비용 | 비고 |
|----------|------|:---------:|:---------:|------|
| kakao | 알림톡 | 1,000자 | 8원 | 템플릿 기반, 가장 저렴 |
| sms | SMS/LMS | 2,000자 | 30원(LMS) / 70원(SMS) | 90자 초과 시 자동 LMS 전환 |
| push | 앱 푸시 | 500자 | 무료 | 회원앱 설치 회원만 |

SMS 채널 선택 시 글자수 90자 기준으로 자동 판별:
- 90자 이하 → SMS (70원/건)
- 90자 초과 → LMS (30원/건)

### 2.4 메시지 전송 탭 — 포인트 현황

| 영역 | 내용 |
|------|------|
| StatCard | 보유 포인트: "52,480 P", 약 SMS 2,624건 발송 가능 |
| 발신 설정 | Select: 발신번호 선택 (예: 02-1234-5678) |
| 채널 상태 | StatusBadge: 알림톡 활성(success), SMS 가능(default) |

### 2.5 수동 메시지 발송 폼

#### 2.5.1 수신자 선택

**그룹 빠른 선택 버튼**: 전체 / 활성 회원 / 만료임박 (7일) / PT 회원 / 장기 미출석

- "전체" 클릭 → `useAllMembers = true`, 개별 선택 초기화
- 기타 그룹 클릭 → 토스트 표시 (향후 기능)

**개별 선택**: [대상 검색] 버튼 → 수신자 검색 모달

#### 2.5.2 수신자 검색 모달 (RecipientModal)

| 영역 | 설명 |
|------|------|
| 헤더 | "수신자 검색" + 선택된 인원수 표시 |
| 검색 | 이름 또는 전화번호 검색 (Supabase members 테이블 조회) |
| 전체 선택 | Checkbox: 현재 필터링된 목록 전체 선택/해제 |
| 회원 목록 | 이름, 전화번호, membershipType 표시. 체크박스 개별 토글 |
| 하단 | [취소] [선택 완료 (N명)] 버튼 |

#### 2.5.3 발송 채널 선택

3개 라디오 카드 (kakao / sms / push):
- 아이콘 + 라벨 + 건당 비용 표시
- 선택 시 border-primary, bg-primary-light 강조

#### 2.5.4 예상 비용 계산

```
예상 비용 = 수신자 수 * 건당 비용
```

실시간 표시: `{smsTypeLabel} {effectiveCost}원/건 · 수신자 {recipientCount}명` → `예상 발송 비용: {totalCost}원`

#### 2.5.5 메시지 본문 에디터

| 항목 | 설명 |
|------|------|
| 글자수 표시 | `{현재} / {최대}자` 실시간 카운터 |
| 글자수 색상 | 90% 이상 amber, 초과 시 red |
| 글자수 게이지 | 프로그레스 바 (accent → amber → error) |
| 치환 변수 버튼 | `#{이름}`, `#{만료일}`, `#{상품명}`, `#{잔여횟수}`, `#{센터명}` |

#### 2.5.6 예약 발송

- Checkbox: "예약 발송" 체크 시 `datetime-local` 입력 필드 노출
- 체크 해제 시 즉시 발송

#### 2.5.7 발송 버튼 & 미리보기

**발송 버튼 비활성화 조건**:
- 메시지 내용이 비어있음
- 글자수 초과 (`isOverLimit`)
- 수신자 미선택 (`!useAllMembers && recipients.length === 0`)
- 발송 진행 중 (`isSending`)

**발송 흐름**:
1. [메시지 발송] 클릭
2. 내용 비어있으면 toast.warning
3. PreviewModal 표시 (수신자 수, 채널, 메시지 내용 확인)
4. [발송하기] 확인 → messages 테이블 INSERT
5. 성공 토스트 → 폼 초기화 → 이력 갱신

#### 2.5.8 발송 미리보기 모달 (PreviewModal)

| 영역 | 내용 |
|------|------|
| 헤더 | "발송 미리보기" |
| 정보 | 채널 배지 + 수신자 수 |
| 본문 | 메시지 내용 (whitespace-pre-wrap) |
| 하단 | [취소] [발송하기] |

### 2.6 자동 알림 설정 미리보기 (메시지 발송 화면 우측)

메시지 발송 탭 우측에 자동 알림 규칙의 ON/OFF 토글 미리보기를 표시한다. [상세 설정] 버튼 클릭 시 `/message/auto-alarm` 페이지로 이동한다.

**고객 관련 5종**: 계약 완료 시, 생일자 고객, 신규 등록 시, 장기 미출석 시, 쿠폰 만료 시
**상품 관련 4종**: 수강권 만료 시, 수강권 만료 전, 회원권 만료 시, 회원권 만료 전

### 2.7 발송 이력 탭

| 컬럼 | key | width | 렌더링 |
|------|-----|-------|--------|
| 발송일시 | sentAt | 160 | YYYY-MM-DD HH:mm 포맷 |
| 유형 | type | 100(center) | StatusBadge(info): KAKAO/SMS/PUSH |
| 수신자 수 | recipients | 100(right) | 배열 길이 표시 |
| 예약일시 | scheduledAt | 140 | YYYY-MM-DD HH:mm 또는 "-" |
| 상태 | status | 100(center) | SENT→완료(녹), SCHEDULED→예약(amber), FAILED→실패(red) |

**기능**: 엑셀 다운로드 지원, 페이지네이션(10건/페이지)

### 2.8 messages 테이블 INSERT 스키마

```typescript
{
  branchId: number;
  type: 'KAKAO' | 'SMS' | 'PUSH';
  title: null;
  content: string;
  recipients: { id: number; name: string; phone: string }[];  // 전체 선택 시 빈 배열
  status: 'SENT' | 'SCHEDULED';
  sentAt: string | null;       // 즉시 발송 시 현재 시각, 예약 시 null
  scheduledAt: string | null;  // 예약 발송 시 예약 시각
}
```

## 3. 자동 알림

> **라우트**: `/message/auto-alarm`
> **소스**: `src/app/(marketing)/message/auto-alarm/page.tsx`
> **유저케이스**: UC-027 (자동 알림 섹션)

### 3.1 화면 개요

회원 이벤트 발생 시 자동으로 메시지를 발송하는 **13종 알림 규칙**을 관리하는 화면이다. 규칙별 ON/OFF 토글, 발송 채널, 발송 시점(N일 전), 메시지 템플릿을 편집할 수 있다.

### 3.2 상단 액션 버튼

| 버튼 | 동작 |
|------|------|
| 메시지 발송 | `/message` 페이지로 이동 |
| 모두 사용 | 전체 규칙 enabled = true 일괄 변경 |
| 설정 추가 | 커스텀 규칙 추가 (id: `custom-{timestamp}`) → 즉시 편집 모달 |

### 3.3 상단 KPI 카드 (3열)

| 카드 | 내용 |
|------|------|
| 발신 번호 | Select: 대표번호 / 개인번호 선택 |
| 보유 포인트 | 125,400 P (약 8,360건 발송 가능) |
| 활성 알림 규칙 | `{enabledCount} / {totalCount}종` + "정상 작동 중" 배지 |

### 3.4 알림 규칙 13종 상세

#### 3.4.1 고객 관련 (7종)

| # | ID | 규칙명 | 설명 | 기본 채널 | 기본 상태 | N값 입력 | 기본 N값 | 메시지 템플릿 |
|---|-----|--------|------|:---------:|:---------:|:--------:|:--------:|--------------|
| 1 | expire-d7 | 만료 D-7 알림 | 회원권 만료 7일 전 안내 | 알림톡 | ON | O | 7일 전 | "안녕하세요 {이름}님! 회원권이 7일 후 만료됩니다. 재등록 시 특별 혜택을 드립니다." |
| 2 | expire-d3 | 만료 D-3 알림 | 회원권 만료 3일 전 안내 | 알림톡 | ON | O | 3일 전 | "안녕하세요 {이름}님! 회원권이 3일 후 만료됩니다." |
| 3 | expire-d1 | 만료 D-1 알림 | 회원권 만료 1일 전 최종 안내 | SMS | OFF | O | 1일 전 | "내일 {이름}님의 회원권이 만료됩니다." |

#### 4.4.3 쿠폰 상태 자동 계산 로직

```
if (!isActive)                                     → 'inactive' (비활성)
if (validUntil && getDaysUntil(validUntil) < 0)    → 'expired' (만료)
if (totalIssued > 0 && totalUsed >= totalIssued)   → 'exhausted' (소진)
else                                                → 'active' (활성)
```

**상태 배지 표시**:

| computedStatus | 라벨 | variant | 추가 표시 |
|----------------|------|---------|-----------|
| active | 활성 | success(dot) | D-7 이내면 amber "D-{n}" 추가 |
| expired | 만료 | default(dot) | - |
| exhausted | 소진 | warning(dot) | - |
| inactive | 비활성 | default(dot) | - |

#### 4.4.4 삭제 제한

`totalIssued > 0`인 쿠폰은 삭제 버튼 비활성화 (disabled). 삭제 시 실제 DELETE가 아닌 `isActive = false` 소프트 삭제.

## 6. 전자계약 위자드

> **라우트**: `/contracts/new`
> **소스**: `src/app/(marketing)/contracts/new/page.tsx`
> **유저케이스**: A-1

### 6.1 화면 개요

회원 선택부터 상품 결제, 전자서명까지 **5단계 위자드**로 전자계약을 진행하는 화면이다. 각 단계별 유효성 검증을 통과해야 다음 단계로 이동할 수 있다.

### 6.2 위자드 5단계

| Step | ID | 이름 | 핵심 기능 |
|:----:|:--:|------|-----------|
| 1 | 회원 선택 | 회원 선택 | 회원 검색 → 테이블에서 선택 |
| 2 | 상품 선택 | 상품 선택 | 카테고리별 상품 카드 → 장바구니 |
| 3 | 기간/금액 | 기간/금액 | 계약 기간 + 할인 + 담당자 설정 |
| 4 | 결제 | 결제 | 결제 수단 선택 + 혼합 결제 |
| 5 | 확인 | 확인 | 계약 요약 + 전자서명 |

### 6.3 스텝 인디케이터

**모바일**: 컴팩트 표시 — 현재 스텝 번호 + 프로그레스 바
**데스크톱**: 풀 스텝 표시 — 5개 원형 번호 + 연결 라인, 완료 단계는 accent + CheckCircle2

### 6.4 Step 1: 회원 선택

| 요소 | 설명 |
|------|------|
| 검색 | 이름 또는 전화번호 검색 (클라이언트 필터링) |
| 신규 등록 | [신규 회원 등록] 링크 → `/members/new` 이동 |
| 테이블 컬럼 | 이름, 연락처, 상태(StatusBadge), 보유 상품, [선택] 버튼 |
| 상태 배지 | ACTIVE→활성(success), EXPIRED→만료(error), HOLDING→홀딩(warning), INACTIVE→미등록(default), SUSPENDED→정지(warning) |
| 선택 확인 | 녹색 확인 카드: "{이름} ({연락처}) 회원이 선택되었습니다." |
| 자동 스크롤 | UX-12: 회원 선택 후 확인 영역으로 smooth scroll |

**유효성 검증**: 회원 미선택 시 "계약 대상 회원을 선택해주세요." 에러 표시

### 6.5 Step 2: 상품 선택

#### 6.5.1 카테고리 탭

| 탭 key | 라벨 | DB category 매핑 |
|--------|------|-------------------|
| facility | 시설이용 | MEMBERSHIP |
| pt | 1:1수업 | PT |
| gx | 그룹수업 | GX |
| option | 옵션 | 그 외 |

#### 6.5.2 상품 카드

각 상품을 카드 형태로 표시 (2열 그리드). 클릭 시 장바구니에 추가. 이미 추가된 상품은 accent 강조 + CheckCircle2 아이콘 (재선택 불가).

카드 구성: 상품명(볼드) + 기간({N}일) + 가격(formatKRW, primary 볼드)

#### 6.5.3 장바구니 (우측 사이드)

- sticky 패널 (top-xl)
- 선택 상품 목록: 상품명 + 가격 + [삭제] 버튼
- 하단: 총 합계 금액

**유효성 검증**: 상품 0개 시 "최소 한 개의 상품을 선택해주세요." 에러 표시

### 6.6 Step 3: 기간/금액

#### 6.6.1 계약 기간 (FormSection, 2열)

| 필드 | 타입 | 필수 | 기본값 | 비고 |
|------|------|:---:|--------|------|
| 시작일 | Input(date) | * | 오늘 | - |
| 종료일 | Input(date) | - | 자동 계산 | readOnly, `시작일 + max(상품기간) + 서비스일수` |
| 서비스 일수 추가 | Input(number) | - | 0 | 추가 서비스 일수 |

#### 6.6.2 할인 설정 (FormSection, 2열)

| 필드 | 타입 | 선택지/범위 | 비고 |
|------|------|-------------|------|
| 할인 유형 | Select | 없음 / 재등록 / 신규 / 이벤트 / 관리자 재량 | - |
| 할인율 | Input(number) % | 0~50 | 최대 50%, 초과 시 에러 |
| 할인 사유 | Input(text) | - | 할인 유형 선택 시 활성화 |

**할인 계산**:
```
discountAmount = Math.round(totalPrice * (discountRate / 100))
finalPrice = totalPrice - discountAmount
```

할인 적용 시 원가/할인/최종가 요약 카드 표시.

#### 6.6.3 실적 담당자

Select: 직원 목록에서 선택 (staff 테이블 조회). 계약 실적 귀속 직원.

#### 6.6.4 특약 및 메모

Textarea(4행): 계약 시 별도 협의 내용.

**유효성 검증**: 시작일 미입력 시 "계약 시작일을 입력해주세요." 에러 표시

### 6.7 Step 4: 결제

#### 6.7.1 결제 수단 선택 (복수 선택 가능)

| 수단 id | 라벨 | 아이콘 |
|---------|------|--------|
| card | 카드 | CreditCard |
| cash | 현금 | Banknote |
| mileage | 마일리지 | Coins |
| transfer | 계좌이체 | ArrowRightLeft |

복수 선택 시 각 수단별 금액 입력 필드 표시. 금액 합계 검증:
- 일치: "금액이 일치합니다" (green)
- 불일치: "금액 합계가 일치하지 않습니다" (red)

#### 6.7.2 결제 금액 요약

| 항목 | 표시 |
|------|------|
| 상품 합계 | formatKRW(totalPrice) |
| 할인 | - formatKRW(discountAmount) (red) |
| 최종 결제 금액 | formatKRW(finalPrice) (Heading-1, primary 볼드) |

#### 6.7.3 결제 실행

[결제 실행하기] 버튼 → Step 5로 이동

### 6.8 Step 5: 확인 & 전자서명

#### 6.8.1 계약 내용 요약

| 섹션 | 표시 항목 |
|------|-----------|
| 회원 정보 | 이름, 연락처 |
| 계약 조건 | 시작일, 종료일, 결제 수단 |
| 상품 목록 | 상품명 + 가격 목록 |
| 금액 내역 | 원가 합계, 할인(유형+율%), 최종 결제 금액 |

#### 6.8.2 전자서명 (SignaturePad)

| 서명 | 설명 |
|------|------|
| 고객 서명 | SignaturePad (height: 180px). 서명 완료 시 "서명 완료" 체크 표시 |
| 센터장 서명 | SignaturePad (height: 180px). 서명 완료 시 "서명 완료" 체크 표시 |

**서명 유효성**: 양쪽 서명 모두 필수. 미완료 시 "고객 서명과 센터장 서명을 모두 완료해주세요." 에러

#### 6.8.3 계약 완료 처리 흐름

```
1. 서명 유효성 검사 (고객 + 센터장 모두 필수)
2. 중복 결제 확인 (checkDuplicatePayment)
   - 중복 감지 시 → ConfirmDialog: "유사한 결제가 이미 존재합니다. 계속하시겠습니까?"
3. 서명 이미지 업로드 (Supabase Storage: contracts/{branchId}/{contractId}_{type}_{timestamp}.png)
4. contracts 테이블 INSERT
5. sales 테이블 INSERT (매출 기록)
6. updateMembershipPeriod() 호출 (회원권 시작/종료일 갱신 + 상태 ACTIVE)
7. accruePoints() 호출 (마일리지 자동 적립, 결제금액의 1%)
   - 단, 마일리지 단독 결제가 아닐 때만 적립
8. 성공 토스트 → 회원 상세 페이지로 이동
```

#### 6.8.4 contracts 테이블 INSERT 스키마

```typescript
{
  branchId: number;
  memberId: number;
  memberName: string;
  productName: string | null;        // 쉼표 구분 상품명
  amount: number;                     // finalPrice
  startDate: string | null;           // ISO
  endDate: string | null;             // ISO (computedEndDate)
  signatureUrl: string;               // 고객 서명 Storage URL
  managerSignatureUrl: string;        // 센터장 서명 Storage URL
  signedAt: string;                   // 현재 시각 ISO
  status: '서명완료';
  createdAt: string;
}
```

#### 6.8.5 sales 테이블 INSERT 스키마

```typescript
{
  branchId: number;
  memberId: number;
  memberName: string;
  productName: string | null;
  type: '이용권';
  amount: number;                     // finalPrice
  originalPrice: number;              // totalPrice
  discountPrice: number;              // discountAmount
  paymentMethod: string;              // 'CARD+CASH' 등 '+' 구분
  saleDate: string;                   // 현재 시각 ISO
  status: 'COMPLETED';
  memo: string | null;                // discountReason
  staffId: number | null;
  staffName: string | null;
}
```
