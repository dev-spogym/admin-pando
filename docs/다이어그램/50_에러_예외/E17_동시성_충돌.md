# E17 — 동시성 충돌

## 1. 개요

| 항목 | 내용 |
|------|------|
| 에러코드 | E409101 (동시 수정 충돌) |
| HTTP | 409 Conflict |
| 발생 모듈 | 회원, 수업, 이용권, 락커 |
| 영향 화면 | 동시 편집이 발생하는 모든 화면 |

## 2. 발생 조건

- 두 명 이상이 동일 리소스를 동시에 수정
- Optimistic Locking 실패 (version 불일치)
- 락 획득 실패 (DB row lock 경합)
- 크론 자동화와 수동 처리 충돌

## 3. 다이어그램

```mermaid
flowchart TD
    USER_A([사용자 A 수정 요청]) --> LOCK_CHECK{낙관적 잠금<br/>version 확인}
    USER_B([사용자 B 동시 수정]) --> LOCK_CHECK

    LOCK_CHECK -->|version 일치|[DB 업데이트<br/>version +1]
    LOCK_CHECK -->|version 불일치|CONFLICT[E409101<br/>동시 수정 충돌]

     -->|성공|RESP_OK[200 저장 완료]
     -->|데드락|DEADLOCK[DB 데드락 감지]

    DEADLOCK --> RETRY{재시도?}
    RETRY --> LOCK_CHECK
    RETRY -->|초과|SERVER_ERR[E500001 서버 오류]

    CONFLICT --> CONFLICT_MODAL[충돌 안내 모달<br/>최신 데이터 제시]
    CONFLICT_MODAL -->|덮어쓰기|FORCE_UPDATE[강제 저장<br/>내 변경사항 적용]
    CONFLICT_MODAL -->|취소|RELOAD[최신 데이터 로드<br/>내 변경 취소]
    CONFLICT_MODAL -->|비교 후 결정|DIFF_VIEW[변경사항 비교 뷰<br/>항목별 선택]

    FORCE_UPDATE --> RESP_OK
    DIFF_VIEW -->|저장|LOCK_CHECK
```

## 4. 복구/재시도 전략

| 상황 | 전략 |
|------|------|
| Optimistic Lock 실패 | 충돌 모달, 사용자가 덮어쓰기/취소 선택 |
| 데드락 | 자동 재시도 최대 3회 |
| 재시도 초과 | E500001 반환, 페이지 새로고침 유도 |
| 크론 충돌 | 크론 처리 우선, 수동 처리는 재시도 |

## 5. 사용자 노출 메시지

| 상황 | 메시지 |
|------|--------|
| 충돌 감지 | "다른 사용자가 동시에 이 데이터를 수정했습니다. 어떻게 처리하시겠습니까?" |
| 덮어쓰기 확인 | "내 변경사항으로 덮어쓰시겠습니까?" |
