# X15 — 공지사항 발행 → 전지점 전파 → 읽음 집계

## 1. 시나리오 개요

본사 관리자가 전체 지점 대상 공지사항 발행 → 각 지점 직원에게 알림 전파 → 직원들이 공지 확인 → 읽음율 집계하는 시나리오.

| 항목 | 내용 |
|------|------|
| 트리거 | 본사 관리자의 공지 발행 |
| 종료 조건 | 공지 발행 + 전 직원 알림 발송 + 읽음 집계 |
| 참여 도메인 | 설정관리(D9), 본사관리(D10), 직원관리(D7) |

## 2. 전제조건

- 본사 관리자 계정 로그인 상태 (`primary` 또는 `owner` 권한)
- 각 지점 직원이 시스템에 등록되어 있음
- 알림 수신 설정 활성화 상태

## 3. 참여 액터

| 액터 | 설명 |
|------|------|
| 본사관리자 | 공지 작성 및 발행 권한 |
| CRM API | FitGenie CRM 백엔드 |
| DB | 데이터베이스 |
| 알림서비스 | 앱 푸시/SMS/이메일 발송 |
| 직원 | 공지 수신 및 확인 |
| 스케줄러 | 읽음율 집계 크론잡 |

## 4. 시퀀스 다이어그램

```mermaid

    actor HQ as 본사관리자
    participant API as CRM API
    participant DB as DB
    participant N as 알림서비스
    actor E as 직원
    participant SCH as 스케줄러

    Note over HQ,API: [공지 작성 및 발행]
    HQ->>API: 1. POST /notices {title, content, target=ALL_BRANCHES, priority} %% API->>DB: 2. notices {status=DRAFT, author_id, created_at}
    DB-->>API: 3. notice_id
    HQ->>API: 4. POST %% API->>DB: 5. notices SET status=PUBLISHED, published_at=NOW(동작)
    API->>DB: 6. employees WHERE active=true (전 지점 직원 목록)
    DB-->>API: 7. 직원 목록

    API->>DB: 8. notice_reads {notice_id, employee_id, status=UNREAD} (전 직원) %% DB-->>API: 9. OK

    loop 각 직원별 알림 발송
        API->>N: 10. 공지 알림 트리거 {employee_id, notice_id} %% N-->>E: 11. 앱 푸시/SMS — "새 공지: {title}"
    end
    API-->>HQ: 12. 200 OK {notice_id, target_count}
    Note over HQ: SCR-085 공지사항 관리 화면

    Note over E,API: [직원 공지 확인]
    E->>API: 13. GET %% API->>DB: 14. notice WHERE id=?
    DB-->>API: 15. 공지 내용
    API->>DB: 16. notice_reads SET status=READ, read_at=NOW(동작) WHERE notice_id=? AND employee_id=? %% DB-->>API: 17. OK
    API-->>E: 18. 공지 내용 + 200 OK

    Note over SCH,DB: [읽음율 집계 — 주기적]
    SCH->>API: 19. 공지 읽음율 집계 트리거 %% API->>DB: 20. (*) total, (CASE WHEN status=READ) read FROM notice_reads WHERE notice_id=?
    DB-->>API: 21. {total, read, unread}
    API->>DB: 22. notices SET read_rate = read/total*100
    DB-->>API: 23. OK
    API-->>SCH: 24. 집계 완료

    Note over HQ,API: [읽음 현황 확인]
    HQ->>API: 25. GET %% API-->>HQ: 26. {read_rate, unread_employees}
    Note over HQ: SCR-085 공지 상세 — 읽음 현황 표시
```

## 5. 주요 메시지 설명

| 번호 | 메시지 | 설명 |
|------|--------|------|
| 4 | POST /publish | 초안(DRAFT)을 발행(PUBLISHED)으로 상태 변경 |
| 8 | notice_reads | 발행 시점에 전 직원의 읽음 레코드 UNREAD로 초기화 |
| 16 | notice_reads | 조회 시점에 READ로 변경. 중복 조회 시 read_at 갱신 안 함 |
| 22 | notices read_rate | 퍼센트로 저장. 대시보드 집계용 |

## 6. 예외/분기

| 상황 | 처리 방법 |
|------|-----------|
| 특정 지점 대상 발행 | target=BRANCH_IDS 파라미터로 대상 지정 |
| 발행 취소/수정 | PUBLISHED → DRAFT 복귀 가능 (발행 취소). 수정 시 재발행 필요 |
| 알림 발송 실패 | 개별 실패 로그 기록, 재발송 큐 등록 |
| 직원 퇴사 후 공지 | active=false 직원은 발송 제외 |
