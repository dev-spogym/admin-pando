- `POST /auth/logout` 또는 Supabase `auth.signOut()` 진행 중

## 상태 고유 인터랙션
- 모든 닫기 수단 차단(일시적)
- Tab 키는 순환하지만 버튼이 disabled 이므로 실효 없음
- 이 상태는 보통 300ms~2s 지속, 2s 타임아웃 후 강제 진행

## 역할별 차이
- 없음

## 상태 고유 비즈니스 룰 델타
- **실행 순서**(useLogout 훅 내부):
  1. `supabase.auth.signOut()` (Promise.race + 2s 타임아웃)
  2. `useAuthStore.clear()` + `useBranchStore.clear()`
  3. `localStorage` 민감값 제거(`authToken`, `refreshToken`, `branchId`), `sessionStorage.clear()`
  4. `router.replace('/login?logout=true')`
- 서버 로그아웃 실패 시 클라이언트 클리어는 강제 진행
- 실패 케이스(네트워크/500) 에서도 사용자 경험상 로그인 페이지로 보내는 것이 안전(재인증 유도)
- 선택적 실패 처리: 클라이언트 클리어 성공 후 `router.replace` 가 실패한 극단적 경우만 토스트 + 모달 재오픈(로그인 버튼 재활성)

## 에지 케이스
- **Supabase signOut 2s 타임아웃**: 클라이언트 클리어 강제 진행, `/login?logout=true&warn=1` 로 경고 쿼리 추가(선택)
- **이미 `/login` 경로**: `router.replace('/login')` 호출은 그대로 동작, 로그인 폼 리렌더
- **동시 다른 탭 로그아웃**: `storage` 이벤트로 다른 탭도 `/login` 이동
- **네트워크 완전 단절**: 2s 후 클리어, 이후 온라인 복구 시 서버 세션은 TTL로 자연 소멸

## 🧩 바이브코딩 프롬프트 (델타)
```
상속: 00-기본화면.md §14 프롬프트 전체 적용
상태: logout-confirm-processing (처리 중)

UI 변경점:
- 버튼 aria-busy=true, disabled
- Primary 버튼 텍스트/아이콘 스위칭:
  {processing && <Loader2 className="size-4 animate-spin" aria-hidden />}
  {processing ? '로그아웃 중...' : '로그아웃'}
- 모달 aria-busy 속성
- ESC/배경 클릭 차단:
  onClick={(e) => { if (e.target === e.currentTarget && !processing) onClose(); }}
  useEffect(() => {
    const onKey = (e: KeyboardEvent) => {
      if (e.key === 'Escape' && !processing) onClose();
    };
    ...
  }, [processing]);

handleConfirm 핸들러:
async function handleConfirm() {
  if (processing) return;
  setProcessing(true);
  try {
    await doLogout();        // useLogout 훅
    // 라우팅 실패 폴백
  } catch (e) {
    toast.error('로그아웃에 실패했습니다. 다시 시도해 주세요');
    setProcessing(false);
  }
}
```
