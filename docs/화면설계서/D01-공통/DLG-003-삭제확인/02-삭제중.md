## 상태 고유 인터랙션
- 모든 상호작용 차단 (중복 삭제 방지)
- ESC `preventDefault`
- 배경 클릭 무시
- 일반적으로 200~800ms 유지

## 역할별 차이
- 없음

## 상태 고유 비즈니스 룰 델타
- 클라이언트 타임아웃 10초 → 실패 처리 → `04-삭제실패` 전환, 토스트 "요청 시간이 초과되었습니다"
- **낙관적 업데이트**: 부모가 `onMutate` 에서 목록 제거 → 실패 시 `onError` 에서 롤백
- 일괄 삭제 진행 중 취소 불가 (서버 트랜잭션 설계 기준)

## 에지 케이스
- **서버가 202 Accepted 반환**: 비동기 작업 큐로 처리. 현재는 200/204 가정
- **응답 지연(>5s)**: 로딩 중 UI 유지. 유저가 다른 탭 이동해도 fetch 는 계속
- **네트워크 단절 후 재연결**: 응답 미수신 → 타임아웃 → `04` 전환
- **동시 삭제(다른 사용자)**: 2번째 요청은 404 → `04` 에서 "이미 삭제된 항목입니다" 안내

## 🧩 바이브코딩 프롬프트 (델타)
```
상속: 00-기본화면.md §14 프롬프트 전체 적용
상태: delete-confirm-deleting

핵심 변경:
- loading=true → Primary 버튼 스피너 + "처리 중..."
- ESC 차단: useEffect(() => { const onKey = (e) => { if (e.key==='Escape') e.preventDefault(); }; ... }, [loading])
- onClick 가드: onClick={(e) => { if (e.target === e.currentTarget && !loading) onCancel(); }}
- useMutation 에서 mutateAsync + timeout race:
  async function handleDelete() {
    try {
      await Promise.race([
        delMutation.mutateAsync(id),
        new Promise((_,rej) => setTimeout(()=>rej(new Error('timeout')), 10000)),
      ]);
    } catch (e) { /* onError 분기 */ }
  }
```
