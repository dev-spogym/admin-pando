# 스프레드시트 구조 대비 리포트/KPI 갭 분석

기준일: 2026-04-13  
비교 대상: 지점 원장 파일, AppScript 취합, 본사 취합 파일까지 포함한 스프레드시트 운영 구조  
비교 기준: 현재 `main` 브랜치의 실제 화면/코드 구현 상태

## 1. 판단 기준

- `대체 구현`: 파일 구조는 없지만 웹 화면/DB로 핵심 목적은 수행 가능
- `부분 구현`: 조회나 일부 다운로드는 가능하지만, 저장형 보고서/자동화/취합 구조가 빠져 있음
- `미구현`: 현재 사이트에서 해당 흐름 자체를 찾지 못함
- `구조 차이`: 스프레드시트식 파일/폴더/탭 운영 자체를 웹앱으로 대체한 항목

## 2. 한눈에 결론

현재 사이트는 스프레드시트의 `원본 파일 -> AppScript 취합 -> 본사 취합 파일` 구조가 아니라, `DB 실시간 조회 -> 화면 표시 -> 일부 다운로드` 구조에 가깝습니다.

그래서 아래는 어느 정도 되어 있습니다.

- 전 지점 KPI 요약 조회
- 지점 비교 리포트
- 일/주/월 리포트 UI
- 월 목표 기반 KPI 화면

반대로 아래는 아직 부족하거나 미구현입니다.

- 본사 보고서 취합 컨트롤 타워
- 저장형 일간/주간/월간 리포트와 취합 이력
- PT/필라테스/골프 부서 OT 중심 보고 체계
- 서버 배치/스케줄러 기반 자동 생성
- 리포트 이메일 발송
- 스프레드시트처럼 다중 탭 워크북을 만드는 방식의 보고서 산출

## 5. 구조 차이로 보는 항목

아래는 "현재 사이트에 없다"는 점에서는 맞지만, 단순 미구현이라기보다 스프레드시트 운영 방식을 웹앱 구조로 바꾸면서 사라진 항목입니다.

| 항목 | 설명 | 판단 |
| --- | --- | --- |
| 드라이브 폴더 구조 | `{지점}/{연도}/{월}` 폴더와 파일 복제 중심 운영 | 구조 차이 |
| 월별 원본 파일 생성 | `매출종합`, `신규통합 DB`, `재등록 DB`, `OT관리`를 월마다 새 파일로 생성 | 구조 차이 |
| AppScript 버튼 복사 흐름 | 셀 데이터를 복사해 보고 파일을 만드는 방식 | 구조 차이 |
| 보고 파일 내 탭 구조 | `목표/달성/1~6주차`, `일별 탭 31개` 같은 탭 중심 산출물 | 구조 차이 |

즉, 이 부분은 "없어서 문제"라기보다 "지금 제품이 같은 목적을 다른 구조로 풀고 있는가"를 따로 판단해야 합니다.

## 6. 개발 우선순위 제안

스프레드시트 운영과 가장 유사한 본사 관리 체계를 만들려면 아래 순서가 효율적입니다.

1. `본사 보고서 취합`에 해당하는 저장형 리포트 엔티티부터 추가
   - 리포트 종류, 기준 기간, 대상 지점, 생성 상태, 생성 시각, 생성 결과 파일/데이터 snapshot
2. 지점 리포트를 화면 상태가 아니라 서버에서 생성하도록 변경
   - 일간/주간/월간 보고서를 DB 또는 파일 스토리지에 저장
3. 본사 취합 화면 추가
   - 일일/주간/월간 취합, 전지점매출현황, PT/필라/골프 취합을 한 화면에서 트리거
4. 이메일 발송과 스케줄러 연결
   - 수동 생성 후 발송
   - 이후 정기 배치로 확장

## 7. 요약

현재 사이트는 `조회형 KPI 대시보드`와 `간단한 리포트 UI`는 있지만, 스프레드시트가 갖고 있는 핵심 강점인 `월별 산출물 보관`, `본사 취합`, `부서별 OT 리포트`, `배치 자동화`, `발송`은 아직 구현되지 않았습니다.

따라서 "스프레드시트 운영을 웹으로 완전히 옮긴 상태"라고 보기는 어렵고, 현재는 그중 `실시간 조회` 영역만 먼저 올라와 있는 상태로 보는 것이 맞습니다.
