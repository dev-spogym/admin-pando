## 1. 회원 목록 (`/members`)

### A. 화면 개요

| 항목 | 내용 |
|------|------|
| 화면명 | 회원 목록 |
| URL | `/members` (내부 pageId: 967) |
| 접근 가능 역할 | primary, owner, manager, fc, staff |
| 진입 경로 | 좌측 사이드바 > 회원관리 > 회원 목록 |
| 목적 | 센터에 등록된 전체 회원을 조회하고, 필터/검색/정렬을 통해 원하는 회원을 찾아 상세 페이지로 이동하거나 일괄 작업(상태 변경, 메시지 발송, 출석 처리, 관심회원 등록)을 수행한다. |
| 사용 시나리오 | FC가 오늘 만료 예정 회원을 검색하여 리텐션 상담 / 매니저가 30일 이상 미방문 회원에게 일괄 메시지 발송 / 센터장이 전체 회원 엑셀 다운로드 |

### I. 필터/검색/정렬 상세

**검색:**
- 대상 필드: `name` (이름), `phone` (연락처) — Supabase `or` 쿼리로 `ilike` 검색
- 디바운스: 300ms (`setTimeout` 기반)
- placeholder: "회원명, 연락처 검색..."

**필터 (SearchFilter 컴포넌트):**

| # | key | label | type | 옵션 |
|---|-----|-------|------|------|
| 1 | product | 계약상품 | select | 전체, PT, 헬스, 요가, 필라테스 |
| 2 | gender | 성별 | select | 전체, 남, 여 |
| 3 | expiryDate | 최종만료일 | dateRange | 날짜 범위 선택 |
| 4 | visitDate | 최근방문일 | dateRange | 날짜 범위 선택 |

**추가 필터 행:**

| # | 필터명 | 타입 | 옵션/동작 |
|---|--------|------|-----------|
| 1 | 관심회원만 | checkbox | `onlyFavorite` 토글. 체크 시 isFavorite=true만 조회 |
| 2 | 미방문 | Select 드롭다운 | 전체(0), 7일 초과(7), 14일 초과(14), 30일 초과(30). lastVisitAt < (현재 - N일) |
| 3 | 회원구분 | Select 드롭다운 | 전체, 일반, 기명법인, 무기명법인 |
| 4 | 유입경로 | Select 드롭다운 | 전체, 홈페이지, 지인소개, 네이버, 인스타, 기타 |
| 5 | 만료 숨기기 | checkbox | `hideExpired` 토글. 체크 시 status='EXPIRED' 레코드를 클라이언트에서 필터링 |

**정렬:**
- 정렬 가능 컬럼: name (회원명), membershipExpiry (만료일), registeredAt (등록일)
- 기본 정렬: `createdAt` 내림차순 (최신순)
- 서버사이드 정렬: Supabase `.order(orderCol, { ascending })` 사용

**페이지네이션:**
- 기본 페이지 크기: 20
- 페이지 크기 옵션: [20, 50, 100]
- 서버사이드 페이지네이션: `.range(from, to)` 사용

### D. 전화번호 중복확인 로직

1. 사용자가 전화번호를 입력하면 `formatPhone()`으로 자동 포맷 (010-xxxx-xxxx)
2. 번호 변경 시 `phoneChecked`와 `phoneDuplicate` 초기화
3. [중복확인] 버튼 클릭:
   - `trigger("phone")`으로 Zod 검증 수행
   - 검증 통과 시: `supabase.from('members').select('id').eq('branchId', branchId).eq('phone', phone).is('deletedAt', null)` 조회
   - 수정 모드일 때: `.neq('id', urlMemberId)`로 자기 자신 제외
   - 결과 있으면: `phoneDuplicate = true`, 표시: "이미 등록된 번호입니다." (빨간색)
   - 결과 없으면: `phoneDuplicate = false`, 표시: "사용 가능한 번호입니다." (초록색)
4. disabled 조건: `phoneChecked`가 true이거나, phone이 없거나, phone.length < 13

#### 6.1.1 getMembers 필터 적용 순서

```
1. .eq('branchId', branchId)              -- 현재 지점
2. .is('deletedAt', null)                 -- soft delete 제외
3. params.status !== 'all'
   → .eq('status', params.status.toUpperCase())
4. params.search
   → .or(`name.ilike.%${search}%,phone.ilike.%${search}%`)
5. params.gender !== 'all'
   → .eq('gender', gender === 'male' ? 'M' : 'F')
6. params.product !== 'all'
   → .ilike('membershipType', `%${product}%`)
7. params.staffId
   → .eq('staffId', staffId)
8. params.isFavorite === true
   → .eq('isFavorite', true)
9. params.daysNoVisit > 0
   → .lt('lastVisitAt', cutoffDate.toISOString())
10. params.memberType !== 'all'
    → .eq('memberType', memberType)
11. params.referralSource !== 'all'
    → .eq('referralSource', referralSource)
12. .range(from, to)                      -- 페이지네이션
13. .order(orderCol, { ascending })       -- 정렬
```

### 8.2 홀딩 처리 (`startHolding`)

**함수 위치:** `src/lib/businessLogic.ts`

**실행 순서:**

1. `supabase.from('members').update({ status: 'HOLDING' }).eq('id', memberId)` — 상태 변경
2. 현재 membershipExpiry에 holdDays(일)만큼 가산하여 새 만료일 계산
3. `supabase.from('members').update({ membershipExpiry: newExpiry }).eq('id', memberId)` — 만료일 연장
4. `supabase.from('member_holdings').insert({ memberId, startDate: today, endDate: today+holdDays, reason: holdReason, status: 'ACTIVE', createdBy, createdByName })` — 홀딩 이력 생성

**반환값:** `{ success: boolean, message: string }`

**성공 시 message:** "홀딩 처리가 완료되었습니다. 만료일이 {holdDays}일 연장되었습니다."  
**실패 시 message:** Supabase error.message

### 8.3 홀딩 해제 (`endHolding`)

**실행 순서:**

1. `supabase.from('members').update({ status: 'ACTIVE' }).eq('id', memberId)` — 상태 복원
2. member_holdings 테이블에서 해당 회원의 ACTIVE 홀딩 레코드를 COMPLETED로 변경

**반환값:** `{ success: boolean, message: string }`

### 8.6 이관 프로세스

**단계별 상세:**

**Step 1 — 적격성 확인 (`checkTransferEligibility`):**

| 체크 항목 | 데이터 소스 | 판정 |
|-----------|------------|------|
| 미납금 | sale 테이블에서 unpaid > 0 합계 | 합계 > 0이면 block |
| 락커 | locker 테이블에서 memberId 매칭 건 | 건수 > 0이면 block |
| 홀딩 상태 | members.status === 'HOLDING' | true이면 warn |
| PT 잔여 | 계약 중 PT 잔여 횟수 조회 | 잔여 > 0이면 warn |
| 쿠폰 | member_coupons에서 status !== 'USED' 건수 | 건수 > 0이면 info |
| 마일리지 | members.mileage | 항상 ok (테넌트 통합) |

**Step 2 — 판정:**
- blockers 배열 length > 0 → canTransfer = false
- blockers 없고 warnings만 있음 → canTransfer = true + 주의 배너
- 둘 다 없음 → canTransfer = true + "모든 항목 통과" 배너

**Step 3 — 이관 실행 (`transferMember`):**
- `supabase.from('members').update({ branchId: toBranchId }).eq('id', memberId)` — 지점 변경
- 이관 로그 테이블에 레코드 생성 (fromBranchId, toBranchId, transferType, reason, timestamp)
- transferType에 따라 추가 처리:
  - KEEP_TICKET: 이용권 유지 (별도 처리 없음)
  - REFUND_NEW: 환불 프로세스 트리거 + 이용권 초기화
  - EXHAUST_THEN: 이관 예약 상태로 설정 (향후 구현)

### 8.7 전화번호 중복 확인 (`checkDuplicateMember`)

**함수 위치:** `src/lib/businessLogic.ts`

**Supabase 쿼리:**
```
supabase
  .from('members')
  .select('id')
  .eq('branchId', branchId)
  .eq('phone', phone)
  .is('deletedAt', null)
```

**수정 모드:** `.neq('id', currentMemberId)` 추가하여 자기 자신 제외

**반환:** data.length > 0이면 중복 존재

### 8.8 회원 삭제 (Soft Delete)

**Supabase 쿼리:**
```
supabase
  .from('members')
  .update({ deletedAt: new Date().toISOString(), status: 'INACTIVE' })
  .eq('id', memberId)
```

**audit log:** `createAuditLog({ action: AUDIT_ACTIONS.DELETE, targetType: 'member', targetId: id })`

**주의:** 실제 데이터 삭제가 아닌 soft delete. deletedAt 컬럼이 설정되면 모든 목록 쿼리에서 `.is('deletedAt', null)` 조건으로 제외됨.

### 8.9 회원 탈퇴 처리

**Supabase 쿼리:**
```
supabase
  .from('members')
  .update({
    status: 'WITHDRAWN',
    withdrawReason: withdrawReason.trim() || null,
    withdrawnAt: new Date().toISOString()
  })
  .eq('id', memberId)
```

**soft delete와의 차이:**
- 탈퇴: status를 'WITHDRAWN'으로 변경. deletedAt은 설정하지 않음. 목록에서는 status 필터로 제외.
- 삭제: deletedAt을 설정하여 모든 쿼리에서 완전히 제외.

### 9.2 UX 개선 제안

| # | 현재 상태 | 개선 제안 |
|---|-----------|-----------|
| 1 | 회원 상세에 15개 탭이 수평 나열 | 탭 그룹핑 (기본정보/결제/운동/관리) 또는 탭 드롭다운으로 정리하여 가독성 향상 |
| 2 | 상세내역 탭 내 5개 서브탭 | 각 서브탭을 독립 탭으로 승격하거나, 더 명확한 네이밍 적용 |
| 3 | 결제 이력 탭(payment)과 결제내역 탭(payment_detail) 중복 | 하나로 통합. payment_detail이 더 상세하므로 payment 탭 제거 고려 |
| 4 | 체성분 탭(body)과 신체정보 탭(bodyInfo) 데이터 소스 분리 | 두 테이블(bodyComposition, member_body_infos) 통합 또는 역할 명확화 |

| 6 | 회원 목록 페이지 로딩 시 통계 카드와 테이블 동시 로딩 | Skeleton UI 적용으로 로딩 UX 개선 |
| 7 | 모달 닫기 시 ESC 키 지원이 일부 모달에만 적용 | 전체 모달에 ESC 키 닫기 통일 |
