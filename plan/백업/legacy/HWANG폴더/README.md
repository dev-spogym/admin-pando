# FitGenie 정리본 기획서 묶음

> 작성 목적: 기존 산출물을 수정하지 않고, 현재 충돌하는 기획 기준을 퍼블리싱/기획 기준으로 다시 정리한 별도 문서 묶음
> 작성 원칙: 이 폴더 밖의 기존 문서는 읽기 전용으로 간주하며 수정하지 않음
> 적용 범위: 관리자 웹(D01~D11), 회원앱(D12), 자동화/알림 정책, 상태 체계, 권한 체계

---

## 사용 원칙

1. 이 폴더 문서는 "정리된 신규 기준서"로 사용한다.
2. 기존 `docs/기능명세서`, `docs/화면설계서`, `docs/다이어그램`, `docs/회원앱` 문서는 원본 보존용으로 유지한다.
3. 추후 기존 문서를 정비할 때는 반드시 이 폴더 문서를 기준으로 반영한다.
4. 본 문서군은 백엔드 라우터 상세가 아니라 기획/퍼블리싱 기준 정합성에 초점을 둔다.

## 문서 목록

- `01_전사_정합성_정리안.md`
  전체 서비스에서 충돌하던 기획 기준을 한 번에 정리한 상위 기준서
- `02_상태체계_통합안.md`
  리드, 회원, 이용권, 결제 상태를 하나의 체계로 통합한 기준서
- `03_권한체계_통합안.md`
  관리자 웹과 회원앱 역할 체계를 분리해 정리한 기준서
- `04_자동화_만료알림_정책_개정안.md`
  본사가 만료 알림 규칙을 등록하고 지점이 선택 사용하는 운영안
- `05_문서정비_우선순위.md`
  추후 원본 문서를 손볼 때의 정비 순서와 체크리스트
- `운영용_기획문서/`
  실제 운영 기준으로 재구성한 도메인별 운영 문서 세트
- `기능명세서/`
  배포용 기준에 맞춰 재정리한 기능명세서 세트
- `화면설계서/`
  퍼블리싱과 QA 기준으로 사용할 핵심 화면 마스터 세트
- `배포용_완성본_체크리스트.md`
  실제 배포 기준으로 문서 범위와 포함 여부를 점검하는 최종 체크리스트

---

## 핵심 결론

- 리드 상태, 회원 상태, 결제 상태는 현재 문서마다 서로 다른 값을 사용하고 있으므로 통합 기준이 필요하다.
- 관리자 웹 권한과 회원앱 권한은 같은 이름을 공유하더라도 별도 체계로 관리해야 한다.
- 만료 알림은 고정 D-day가 아니라, 본사가 정책 세트를 만들고 지점이 선택 사용하는 구조가 적합하다.
- 퍼블리싱 산출물은 "어떤 상태가 어떤 배지/CTA/빈상태/제한 상태를 보여야 하는가"가 고정되어야 하므로, 상태 체계와 권한 체계를 먼저 정리해야 한다.

---

## 현재 기준

- 이 폴더는 이제 정리 메모 수준이 아니라, 실제 배포용 기준 패키지로 사용한다.
- 원본 `docs`는 계속 보존하고, 기획/퍼블리싱/QA 기준 판단은 이 폴더를 우선한다.
