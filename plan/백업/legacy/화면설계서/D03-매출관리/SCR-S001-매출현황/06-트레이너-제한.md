## 역할별 차이
이 상태 자체가 trainer 전용이므로, 진입 시점에 이미 결정. 다른 역할은 본 파일에 해당 없음.

## 상태 고유 인터랙션
- 담당자 필터 자물쇠 클릭 → 툴팁 "본인 담당만 조회 가능합니다"
- POS 버튼 영역 hover → "POS 권한이 없습니다. 센터장에게 문의하세요" 툴팁(숨김 상태라 툴팁은 선택)
- 행 클릭 → DLG-S001 매출상세 (정상 동작)
- 상세 모달에서 [결제 취소] 버튼 숨김 (trainer는 환불 처리 불가)
- 엑셀 다운로드 시 본인 담당 데이터만 내보냄

## 상태 고유 비즈니스 룰 델타
- **서버 강제**: `WHERE s.staffId = :userId` (JWT claim 기반). 클라 필터 우회 불가.
- **API 호출**: `GET /sales?staffId={userId}&from={}&to={}` (클라에서 명시 전달, 서버는 이중 검증)
- **읽기 전용**: 트레이너는 결제/환불/취소 처리 불가. 상세 모달에서 [영수증 재발행], [회원 상세]만 허용.
- **담당 조건**: `sales.staffId === trainer.id` OR `sales.memberAssignee === trainer.id` (member.assignedTrainerId=trainer.id)
- `delta` 계산(전월대비): 본인 담당 전월 매출 기준
- 집계 탭 (TAB-002/003/005): 본인 담당 한정
- RLS: Supabase 정책으로도 staffId 강제 (`policy: staffId = auth.uid()`)

## 에지 케이스
- 담당 회원이 아직 없는 신규 트레이너 → 0건 → `03-빈상태` (InfoBanner는 유지, 변형 문구 "담당 매출 데이터가 없습니다.")
- 퇴사 예정 트레이너 (status='pending_exit') → 진입 가능하나 [신규 결제] 외 기능 동일
- 담당 이관(회원이 타 트레이너로 이동) 후: 이관 전 매출은 계속 본인 실적으로 표시 (매출 시점 기록)
- 담당자 필터 URL 조작 시 서버가 403 리턴 → `/forbidden`
