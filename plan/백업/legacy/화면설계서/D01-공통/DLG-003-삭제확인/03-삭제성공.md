> 상속: [`00-기본화면.md`](./00-기본화면.md) — 이 문서는 **변경점(델타)**만 기술합니다.

## 상태 진입 조건
- `DELETE /resource/:id` → 204 No Content (또는 200 OK)
- `POST /resource/bulk-delete` → `{ deletedCount, failedIds }` 로 응답 (부분 실패 허용)

## 상태 고유 비주얼 델타
- 이 상태는 **거의 순간적** — 다이얼로그 즉시 언마운트 + 토스트만 노출
- 다이얼로그 본체: 렌더 없음
- 토스트:
  - 단건: `toast.success('{targetName}이(가) 삭제되었습니다.')`
  - 일괄 전체 성공: `toast.success('{N}건 삭제 완료')`
  - 일괄 부분 성공: `toast.warning('{N}건 중 {M}건만 삭제되었습니다')`

## 상태 고유 인터랙션
- 사용자 입력 없음(자동)
- 토스트 자동 소멸(3초)

## 역할별 차이
- 없음. 단 감사로그 기록이 역할/branchId 별로 분리되어 서버에 저장됨.

## 상태 고유 비즈니스 룰 델타
### 컨텍스트별 후속 처리
| 부모 컨텍스트 | 동작 |
|---|---|
| 목록 화면(SCR-010, SCR-060 등) | `queryClient.invalidateQueries(['resource'])` 또는 낙관적 제거 확정. 선택 상태 클리어. |
| 상세 화면(SCR-013 회원상세) | `router.replace(listUrl)` (예: `/members`). 삭제된 상세 화면 유지 불필요. |
| 일괄 삭제 | 실패 ID가 있으면 행만 selectionState 유지, 성공 ID는 선택 해제 |

### 감사로그
- 서버가 `AUDIT.DELETE` 자동 기록 (userId, userName, targetType, targetId, branchId, createdAt)
- 클라이언트는 추가 호출 없음

### 낙관적 업데이트
- 부모가 `onMutate` 에서 이미 목록에서 제거했다면 `onSuccess` 에서는 invalidate 만 호출

## 에지 케이스
- **onSuccess 콜백 내부 에러**: 삭제 자체는 성공이므로 실패로 취급하지 않음. 콘솔 경고만.
- **상세 화면에서 라우팅 실패**: `router.replace` 가 실패하면 `window.location.href` 폴백
- **일괄 삭제 전부 실패**: `deletedCount===0` → `04-삭제실패` 로 치환하여 처리

## 🧩 바이브코딩 프롬프트 (델타)
```
상속: 00-기본화면.md §14 프롬프트 전체 적용
상태: delete-confirm-success

onSuccess 핸들러:
function onSuccess(result) {
  if (Array.isArray(ids)) {
    const { deletedCount, failedIds = [] } = result ?? {};
    if (deletedCount > 0 && failedIds.length === 0) {
      toast.success(`${deletedCount}건 삭제 완료`);
    } else if (deletedCount > 0 && failedIds.length > 0) {
      toast.warning(`${ids.length}건 중 ${deletedCount}건만 삭제되었습니다`);
      // failedIds 행은 selectionState 유지
    } else {
      toast.error('삭제에 실패했습니다');
      return; // 04-삭제실패 대신 이쪽에서 토스트 처리
    }
  } else {
    toast.success(`${targetName}이(가) 삭제되었습니다.`);
  }
  qc.invalidateQueries({ queryKey:['resource'] });
  setOpen(false);
  if (context === 'detail') router.replace(listUrl);
}
```
