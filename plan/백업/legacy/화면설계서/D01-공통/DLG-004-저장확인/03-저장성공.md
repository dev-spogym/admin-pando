## 상태 진입 조건
- `PATCH/POST` → 200/201 응답
- 응답 body 에 최신 데이터 + version/etag 포함

## 상태 고유 비주얼 델타
- 다이얼로그 즉시 닫힘 → 본체 렌더 없음
- 토스트: `toast.success('저장되었습니다')` (3초)
- (옵션) 저장 후 이동이면 경로 전환 직전에 다이얼로그 닫기 + 토스트

## 상태 고유 인터랙션
- 사용자 입력 없음
- 자동 처리

## 역할별 차이
- 동일. 감사로그는 서버에 userId/branchId 포함

## 상태 고유 비즈니스 룰 델타
- **스토어/캐시 갱신**:
  - `queryClient.setQueryData(['resource', id], data)` 또는 `invalidateQueries(['resource'])`
  - 부모 폼 `form.reset(data.data)` → `isDirty=false` 복원
  - 저장된 `etag/version` 갱신(다음 저장에 사용)
- **afterSaveAction 분기**:
  - `stay` (기본): 편집 화면 유지, 토스트만 노출
  - `navigate(destination)`: `router.push(destination)` — DLG-002 "저장 후 이동" 패턴과 연동
- **감사로그**: 서버가 자동 기록
- **재사용**: 같은 폼에서 추가 편집 → `isDirty` 가 다시 true 가 되면 재저장 가능

## 에지 케이스
- **응답 본문에 최신 데이터 없음**: `refetch()` 추가 호출
- **setQueryData 후 invalidate 양쪽 호출**: 중복되지만 안전. invalidate 만 사용해도 OK
- **navigate 대상이 권한 없는 경로**: 목적지 진입 시 서버 403 → ERR-001 이동
- **토스트 dedup**: 연속 저장 성공 시 같은 메시지 중복 방지 (기본 Sonner/Radix Toast는 알아서 처리)

## 🧩 바이브코딩 프롬프트 (델타)
```
상속: 00-기본화면.md §14 프롬프트 전체 적용
상태: save-confirm-success

onSuccess 핸들러:
function onSuccess(res: { data: Resource; version: string; etag?: string }) {
  qc.setQueryData(['settings','center'], res.data);
  setEtag(res.etag ?? res.version);
  form.reset(res.data); // dirty 초기화
  toast.success(successMessage ?? '저장되었습니다');
  setOpen(false);
  if (afterSaveAction?.type === 'navigate') {
    router.push(afterSaveAction.destination);
  }
  // stay 는 추가 동작 없음
}
```
