# X21 — 홀딩 등록 → 자동 만료일 연장 → 해제

## 1. 시나리오 개요

부상/여행 등으로 회원이 이용권 홀딩(일시정지) 요청 → 홀딩 기간만큼 이용권 만료일 자동 연장 → 홀딩 해제 시 이용권 재활성화하는 시나리오.

| 항목 | 내용 |
|------|------|
| 트리거 | 회원의 홀딩 요청 |
| 종료 조건 | 홀딩 해제 + 이용권 만료일 연장 재활성화 |
| 참여 도메인 | 회원관리(D2) |

## 2. 전제조건

- 매니저 계정 로그인 상태
- 활성(ACTIVE) 이용권이 존재
- 홀딩 정책이 설정되어 있음 (최대 홀딩 일수, 최대 횟수)

## 3. 참여 액터

| 액터 | 설명 |
|------|------|
| 회원 | 홀딩 요청자 |
| 매니저 | 홀딩 등록/해제 처리 |
| CRM API | FitGenie CRM 백엔드 |
| DB | 데이터베이스 |
| 스케줄러 | 자동 홀딩 만료 처리 |
| 알림서비스 | 홀딩/해제 알림 |

## 4. 시퀀스 다이어그램

```mermaid
sequenceDiagram
    actor MB as 회원
    actor M as 매니저
    participant API as CRM API
    participant DB as DB
    participant SCH as 스케줄러
    participant N as 알림서비스

    Note over MB,M: [홀딩 등록 요청]
    MB->>M: 1. 홀딩 요청 (사유, 기간) %% E_MB_HOLD_REQUEST_01
    M->>API: 2. GET /members/{id}/hold-check %% E_M_HOLD_CHECK_01
    API->>DB: 3. SELECT membership WHERE active=true, SELECT holding_policy
    DB-->>API: 4. 홀딩 가능 여부 { max_days, used_days, remaining_holds }
    API-->>M: 5. 홀딩 조건 확인 결과
    Note over M: DLG-M003 홀딩 등록 모달

    alt 홀딩 조건 충족
        M->>API: 6. POST /holdings { membership_id, start_date, end_date, reason } %% E_M_CREATE_HOLD_01
        API->>DB: 7. INSERT holdings { membership_id, start_date, end_date, status=ACTIVE }
        API->>DB: 8. UPDATE memberships SET status=ON_HOLD, expiry_date = expiry_date + hold_days %% E_API_EXTEND_EXPIRY_01
        DB-->>API: 9. holding_id, new_expiry_date
        API->>N: 10. 홀딩 등록 알림 %% E_API_HOLD_NOTIFY_01
        N-->>MB: 11. SMS — "홀딩 등록 완료. 만료일이 {new_expiry_date}로 연장됩니다"
        API-->>M: 12. 200 OK { holding_id, new_expiry_date }
        Note over M: SCR-M004 이용권 탭에 홀딩 상태 표시

    else 홀딩 불가 (최대 횟수 초과)
        API-->>M: 13. 400 Bad Request { reason: MAX_HOLD_EXCEEDED } %% E_API_HOLD_BLOCKED_01
        Note over M: 홀딩 불가 안내
    end

    Note over SCH,DB: [자동 홀딩 만료 — 매일 체크]
    SCH->>API: 14. 홀딩 만료 체크 %% E_SCH_HOLD_CHECK_01
    API->>DB: 15. SELECT holdings WHERE end_date = TODAY AND status=ACTIVE
    DB-->>API: 16. 만료 홀딩 목록
    loop 각 홀딩별
        API->>DB: 17. UPDATE holdings SET status=EXPIRED %% E_API_EXPIRE_HOLD_01
        API->>DB: 18. UPDATE memberships SET status=ACTIVE
        API->>N: 19. 자동 홀딩 만료 알림
        N-->>MB: 20. SMS — "홀딩이 자동 해제되었습니다. 이용권이 재활성화되었습니다"
    end

    Note over MB,M: [수동 홀딩 조기 해제]
    MB->>M: 21. 홀딩 조기 해제 요청 %% E_MB_HOLD_RELEASE_01
    Note over M: DLG-M004 홀딩 해제 모달
    M->>API: 22. POST /holdings/{id}/release { actual_end_date } %% E_M_RELEASE_HOLD_01
    API->>DB: 23. UPDATE holdings SET status=RELEASED, actual_end_date=?
    API->>DB: 24. UPDATE memberships SET status=ACTIVE, expiry_date = expiry_date - (end_date - actual_end_date)
    DB-->>API: 25. new_expiry_date
    API->>N: 26. 홀딩 해제 알림
    N-->>MB: 27. SMS — "홀딩이 해제되었습니다. 새 만료일: {new_expiry_date}"
    API-->>M: 28. 200 OK { new_expiry_date }
```

## 5. 주요 메시지 설명

| 번호 | 메시지 | 설명 |
|------|--------|------|
| 8 | UPDATE memberships expiry_date | 홀딩 일수만큼 만료일 연장. 이중 홀딩 시 누적 연장 |
| 24 | 조기 해제 만료일 재계산 | end_date - actual_end_date = 미사용 홀딩 일수. 그만큼 만료일 단축 |
| 17 | 자동 만료 처리 | 매일 스케줄러가 end_date 도달한 홀딩 자동 해제 |

## 6. 예외/분기

| 상황 | 처리 방법 |
|------|-----------|
| 최대 홀딩 일수 초과 | 400 반환, 잔여 가능 일수 안내 |
| 이미 홀딩 중 중복 신청 | 기존 홀딩 종료 후 신규 등록 |
| 이용권 만료 후 홀딩 | 만료 이용권에는 홀딩 불가 |
| 홀딩 중 만료일 도달 | 홀딩 종료 후 만료일 연장 적용되므로 만료 안 됨 |
