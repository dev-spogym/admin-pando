# X29 — 로그인 → 2FA → 세션 생성 → 대시보드

## 1. 시나리오 개요

사용자가 ID/PW 입력 → 1차 인증 → 2FA 코드 입력 → 세션/JWT 발급 → 역할 기반 대시보드 진입까지의 인증 플로우.

| 항목 | 내용 |
|------|------|
| 트리거 | 사용자의 로그인 시도 |
| 종료 조건 | 세션 생성 + 대시보드 진입 |
| 참여 도메인 | 공통(D1) |

## 2. 전제조건

- 계정이 시스템에 등록되어 있음
- 2FA 설정이 활성화된 계정 (SMS OTP 또는 TOTP)
- 이전 세션이 없거나 만료 상태

## 3. 참여 액터

| 액터 | 설명 |
|------|------|
| 사용자 | 로그인 시도자 |
| CRM API | FitGenie CRM 백엔드 |
| DB | 데이터베이스 |
| 인증서버 | JWT 발급 서비스 |
| OTP서비스 | SMS/TOTP OTP 발송 |
| 알림서비스 | 로그인 이상 감지 알림 |

## 4. 시퀀스 다이어그램

```mermaid
sequenceDiagram
    actor U as 사용자
    participant API as CRM API
    participant DB as DB
    participant AUTH as 인증서버
    participant OTP as OTP서비스
    participant N as 알림서비스

    Note over U,API: [1차 인증 — ID/PW]
    U->>API: 1. POST /auth/login { email, password } %% E_U_LOGIN_01
    API->>DB: 2. SELECT users WHERE email=?
    DB-->>API: 3. 사용자 정보 (password_hash, status, role, 2fa_enabled)

    alt 계정 없음
        API-->>U: 4. 401 Unauthorized { message: "이메일 또는 비밀번호가 올바르지 않습니다" } %% E_API_LOGIN_FAIL_ID_01
    else 비밀번호 불일치
        API->>DB: 5. UPDATE users SET login_fail_count = login_fail_count + 1
        API-->>U: 6. 401 Unauthorized %% E_API_LOGIN_FAIL_PW_01
        alt 5회 이상 실패
            API->>DB: 7. UPDATE users SET status=LOCKED %% E_API_LOCK_ACCOUNT_01
            API->>N: 8. 계정 잠금 알림
            N-->>U: 9. SMS/이메일 — "계정이 잠겼습니다. 비밀번호를 재설정해 주세요"
        end
    else 비밀번호 일치
        API->>DB: 10. UPDATE users SET login_fail_count=0
        alt 2FA 활성화
            API->>OTP: 11. OTP 발송 요청 { user_id, method=SMS|TOTP } %% E_API_OTP_SEND_01
            OTP-->>U: 12. OTP 코드 (6자리, 3분 유효)
            API-->>U: 13. 200 OK { requires_2fa: true, session_token }
            Note over U: SCR-100 2FA 입력 화면

            U->>API: 14. POST /auth/verify-2fa { session_token, otp_code } %% E_U_2FA_VERIFY_01
            API->>OTP: 15. OTP 검증 요청
            OTP-->>API: 16. 검증 결과

            alt OTP 일치
                API->>AUTH: 17. JWT 발급 요청 { user_id, role, branch_id } %% E_API_JWT_ISSUE_01
                AUTH-->>API: 18. access_token (1h), refresh_token (7d)
                API->>DB: 19. INSERT sessions { user_id, token, created_at, expires_at }
                DB-->>API: 20. session_id
                API-->>U: 21. 200 OK { access_token, refresh_token, role }
                Note over U: 역할별 대시보드로 리다이렉트
            else OTP 불일치 또는 만료
                API-->>U: 22. 401 Unauthorized { message: "OTP가 올바르지 않습니다" } %% E_API_OTP_FAIL_01
            end
        else 2FA 비활성화
            API->>AUTH: 23. JWT 발급 요청 %% E_API_JWT_NO2FA_01
            AUTH-->>API: 24. access_token, refresh_token
            API->>DB: 25. INSERT sessions
            API-->>U: 26. 200 OK { access_token, role }
        end
    end

    Note over U,API: [대시보드 진입]
    U->>API: 27. GET /dashboard { Authorization: Bearer token } %% E_U_DASHBOARD_01
    API->>AUTH: 28. JWT 검증
    AUTH-->>API: 29. 유효 여부 + payload
    API->>DB: 30. SELECT dashboard_data WHERE branch_id=? AND role=?
    DB-->>API: 31. 역할별 대시보드 데이터
    API-->>U: 32. 200 OK { dashboard }
    Note over U: SCR-101 대시보드 표시
```

## 5. 주요 메시지 설명

| 번호 | 메시지 | 설명 |
|------|--------|------|
| 5 | login_fail_count 증가 | 5회 연속 실패 시 계정 잠금. 잠금 해제는 이메일 인증 또는 관리자 처리 |
| 11 | OTP 발송 | SMS: 6자리 랜덤 코드. TOTP: Google Authenticator 등 앱 코드 |
| 17 | JWT 발급 | payload에 user_id, role, branch_id, exp 포함 |
| 19 | INSERT sessions | 리프레시 토큰 관리. 다중 기기 로그인 이력 추적 |
| 30 | 역할별 대시보드 | primary: 전 지점. manager: 자기 지점. fc: 담당 회원만 |

## 6. 예외/분기

| 상황 | 처리 방법 |
|------|-----------|
| 계정 잠금 | 잠금 해제 이메일 발송. 관리자 수동 해제 가능 |
| OTP 만료 | 재발송 요청 (30초 cooldown) |
| JWT 만료 | refresh_token으로 재발급. refresh도 만료 시 재로그인 |
| 비정상 로그인 감지 | 새로운 IP/기기에서 로그인 시 알림 발송 |
