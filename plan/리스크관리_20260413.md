## 2. 리스크 상세 분석

### 🔴 R-01. 기획 잔여 20% 일정 지연

**What (무엇이 문제인가)**
- 전체 기획의 약 80%가 완료되었으나, 남은 20%는 디테일 조정 및 앱(모바일) 기획 영역
- 양적으로 20%이나, 시간적으로는 비례하지 않음 (회의 중 황프로 언급: "20%가 시간으로 20%가 아닙니다")
- 5월 15일까지 잔여 기획 완료 목표

**How (어떻게 대응하는가)**
- 기획 잔여 항목을 항목별로 리스트업하고 우선순위 재배정
- 주 단위 기획 완료 마일스톤 설정 (4/21, 4/28, 5/5, 5/12)
- 개발에 영향 주는 기획 항목을 먼저 확정하여 개발 착수 가능하도록 분리

**When (언제까지)**
- ~4/14 (월): 잔여 20% 항목 리스트업 + 우선순위 확정
- ~5/15: 전체 기획 완료 마감

### 🔴 R-02. 백엔드 API 미연동 (KPI 센터 · Today Tasks)

**What (무엇이 문제인가)**
- KPI 센터와 Today Tasks 화면은 프론트엔드 구현 완료(✅)이나, 백엔드 연동이 원활하지 않음 (회의 중 심프로 언급)
- 현재 더미 데이터 기반으로 동작 중
- 실 데이터 기반 동작 확인 불가 → 데모/설명 시 신뢰도 하락

**How (어떻게 대응하는가)**
- Supabase 기반 백엔드 API 엔드포인트 정의 및 우선 구현
- KPI 센터: KPI 정의서(엑셀 3개 시트) 기준 데이터 테이블 설계
- Today Tasks: 업무 CRUD API + 상태(긴급/높음/보통) 관리 API
- dailySync 서버 사이드 cron 전환 (기술부채 #2)

**When (언제까지)**
- ~4/18: API 스키마 확정 + Supabase 테이블 생성
- ~4/25: KPI 센터 실데이터 연동 완료
- ~5/2: Today Tasks 백엔드 연동 완료

### 🔴 R-03. 구글시트 ↔ CRM 데이터 정합성

**What (무엇이 문제인가)**
- 현재 스포짐은 구글시트 기반으로 운영 중이며, CRM 대시보드와 구글시트 대시보드 간 데이터 차이 존재
- 대표님 지적: "구글시트 베이스 대시보드랑 이 대시보드랑 차이가 있잖아요"
- KPI 자동 리포트 역시 구글시트 항목과의 오버랩/차이 미정리
- 판도 개발팀 설명 시 이 차이를 명확히 안내하지 않으면 혼란 발생

**How (어떻게 대응하는가)**
- 구글시트 대시보드 항목 vs CRM 대시보드 항목 비교표 작성
- 차이나는 항목에 대해 "CRM에서 신규 추가" / "구글시트에서 이관" / "미반영" 표기
- 판도 개발팀 설명 시 **반드시 먼저 언급**: "기존 구글시트 → 판도 CRM 개발 전환 원칙"
- 대표님 참석하는 4/14(월) 미팅에서 딥하게 논의 예정

**When (언제까지)**
- ~4/14 09:10: 판도 개발팀 인트로 설명 (황프로)
- ~4/14(월): 대표님 참석 미팅에서 구글시트 ↔ CRM 차이 확정
- ~4/18: 비교표 문서화 완료

### 🟡 R-04. KPI 항목 불일치 (엑셀 vs CRM)

**What (무엇이 문제인가)**
- KPI 엑셀 시트 1(본사), 2(지점), 3(FC/트레이너) 내용이 CRM에 유기적으로 반영되었다고 하나, 100% 일치 여부 미확인
- 황프로 언급: "판단을 완벽하게 할 수는 없을 것 같습니다. 다른 건 체크해주셔야 된다"

**How (어떻게 대응하는가)**
- KPI 정의서(`docs/KPI_정의서.md`)와 CRM KPI 센터 화면 항목 1:1 대조표 작성
- 황프로가 판단 가능한 범위 먼저 체크 → 나머지는 대표님/성프로 확인 요청
- 불일치 항목은 우선순위 태깅 후 순차 반영

**When (언제까지)**
- ~4/14(월): 황프로 1차 체크 결과 공유
- ~4/18: 대표님 확인 후 최종 확정

### 🟡 R-05. 회원 중복확인 · 지점 간 이동 처리

**What (무엇이 문제인가)**
- 회원 등록 시 전화번호 중복확인 기능 필요 (DB 기준) — 대표님 확인: "당연히 있어야지"
- 타 지점에서 이동한 회원 케이스 처리 로직 미정의 (예: A지점 → B지점 이사)
- 중복 시 FC가 기존 회원 ID를 찾아줘야 하는 워크플로우 필요

**How (어떻게 대응하는가)**
- 전화번호 중복확인 API 구현 (현재 프론트 P1 GAP #5로 구현 완료, 백엔드 연동 필요)
- 중복 발견 시 시나리오 3가지 정의:
  1. 같은 지점 내 중복 → 등록 차단 + 기존 회원 안내
  2. 다른 지점 중복 → "이관 가능" 알림 + FC 처리 워크플로우
  3. 탈퇴 회원 재등록 → 기존 데이터 연결 여부 선택
- 회원 이관(MemberTransfer.tsx) 기능과 연계

**When (언제까지)**
- ~4/21: 중복 시나리오 기획 확정
- ~4/28: 백엔드 중복확인 API + 이관 연동 구현
